## Проблема изоляции: почему AI не видит корпоративные данные

Главная проблема современных AI-фреймворков — изоляция языковых моделей от данных. Если в проекте используются несколько LLM и несколько источников данных, количество интеграций растет экспоненциально. Три AI-модели и пять источников данных без стандартизации требуют **15** отдельных интеграций. С MCP это количество сокращается до **8**, потому что протокол стандартизирует связку клиент-сервер.

Кроме того, без единого протокола невозможно переиспользовать интеграции для разных моделей без адаптации кода. Каждая смена LLM-провайдера превращается в новый проект.

## MCP как «USB-C для AI»: стандарт вместо костылей

Model Context Protocol — открытый стандарт от Anthropic, представленный в ноябре 2024 года. Vendor-agnostic архитектура: Anthropic выкатил протокол в open source, и от самого Anthropic он не зависит. В компании сравнивают MCP с USB-C: один стандартный разъем вместо десятков кастомных коннекторов.

[ИЛЛЮСТРАЦИЯ: слайд #4 — схема MCP как USB-C для AI]

:::accent
MCP сокращает количество интеграций с экспоненциального (M×N) до линейного (M+N). Потенциальная экономия на этапе разработки — до 50%.
:::

Система MCP состоит из трех компонентов. Хост — любое приложение с поддержкой AI (чат-бот, IDE). Клиент — встроен в хост, управляет соединениями по протоколу MCP. Сервер — отдельный сервис, который предоставляет функции: доступ к БД, веб-поиск, файловую систему, корпоративные базы знаний.

## Пять слоев MCP-сервера: от коммуникации до кеширования

Архитектуру MCP-сервера можно разделить на пять слоев, каждый из которых решает свою задачу.

:::steps
### Коммуникационный слой
Протокол JSON-RPC 2.0 — простой и легковесный протокол для контрактов клиент-сервера.

### Обработчики запросов
Три типа обработчиков: ресурсы (read-only данные, справочники), инструменты (запись, удаление, математические расчеты) и подсказки (промпт-билдеры, которые объясняют клиенту, как общаться с сервером).

### Хранилище контекста
Передача данных между запросами внутри одной тулы, когда нужно обращаться к нескольким источникам последовательно и сохранять состояние.

### Оркестратор сессий
Опциональный компонент для поддержки пользовательских сессий — например, когда нужно хранить историю чата.

### Слои кеширования
Ускорение работы MCP через кеш для read-only ресурсов и повторяющихся запросов.
:::

## Как устроена система с MCP: от пользователя до источника данных

В типовой архитектуре пользователь обращается к фронтенду, который передает запрос на бэкенд. На бэкенде включается основная логика: LLM Router выбирает нужную языковую модель (если их несколько), а MCP Gateway проверяет, какая LLM имеет доступ к какому MCP-серверу. После авторизации запрос уходит на нужный MCP-сервер, который обращается к источнику данных — внутренней БД, внешнему API или другому сервису.

[ИЛЛЮСТРАЦИЯ: слайд #11 — схема архитектуры системы с MCP]

MCP Gateway — опциональный, но критически важный компонент. Здесь закладывается основная логика безопасности и авторизации. Без него любая LLM может обратиться к любому серверу — это недопустимо в продакшене.

Создать собственный MCP-сервер можно буквально в шесть строчек на Python с помощью библиотеки FastMCP, написанной на базе FastAPI. Разумеется, это упрощенный пример — но простота входа налицо.

## Угрозы безопасности: четыре атаки, о которых нужно знать

Основная часть безопасности MCP ложится на разработчиков. Протокол сам по себе не несет встроенных механизмов защиты. Угрозы совместимы с OWASP AI Top 10.

:::columns Основные угрозы MCP
### Tool Poisoning
Вредоносные инструкции в метаданных инструментов, невидимые пользователю. Один зараженный tool компрометирует все сессии.

### Command Injection
Выполнение произвольных системных команд через отсутствие валидации пользовательского ввода.

### Prompt Injection
Скрытые команды в промптах и indirect injection через внешние данные — кеш, историю, веб-страницы.

### Supply Chain
Бэкдоры в сторонних серверах, отсутствие SBOM и фиксации версий зависимостей.
:::

> Один из реальных примеров Supply Chain-атаки — инцидент с LangChain, когда незафиксированная зависимость получила обновление с вредоносным кодом и вывела из строя все приложения на базе библиотеки на несколько часов.

Базовая защита строится на трех принципах: Zero Trust (никогда не доверять, всегда проверять), OAuth 2.1 с PKCE для сервисной аутентификации и MCP Gateway для сканирования и аудита всех обращений.

## Кейс Syngenta: AI-ассистент для фермеров через Telegram

Совместный проект с Syngenta — AI-ассистент, который помогает фермерам получать данные из системы Cropwise прямо в поле через Telegram-бота. Фермер открывает чат и спрашивает, например, историю засева на конкретном поле за прошлый год. Система должна найти ответ среди порядка **40** API-эндпоинтов Cropwise и данных в PostgreSQL.

Запросы бывают двух типов. Известные — предзаданные, когда система точно знает, в какие таблицы обращаться. Неизвестные — произвольные, когда нужно сначала определить, какие данные и из каких таблиц извлечь. Кроме получения данных, ассистент помогает заполнять отчеты.

[ИЛЛЮСТРАЦИЯ: слайд #20 — схема архитектуры кейса Syngenta]

Архитектура проекта включает несколько MCP-серверов. Пользователь общается через UI — Telegram-бот (основной канал) или React-приложение. За UI стоит агент, который координирует работу трех MCP-серверов: антифрод-сервер для проверки валидности запроса, основной сервер с LLM для обращения к API-источникам и сервер для работы с базой данных — составление SQL-запросов, агрегация данных. Отдельно работает Short Memory — слой кеширования, который ускоряет повторные запросы к справочникам.

## SQL-запросы и ретраи: самая сложная часть проекта

Формирование правильных SQL-запросов языковой моделью — самая сложная задача проекта. Приходилось постоянно дорабатывать промпты, добавлять гайдлайны, подключать RAG-модули, чтобы модель корректно составляла запросы, валидировала их и только потом выполняла.

Если запрос возвращает ошибку, срабатывает механизм ретрая: модель получает обратную связь с текстом ошибки и пытается исправить запрос. Лимит — **3** попытки. До третьей попытки доходит, но редко.

:::accent
Правильное формирование SQL-запросов AI-моделью требует постоянной настройки промптов, RAG-модулей и валидации. Это не задача, которая решается «из коробки» — даже с MCP.
:::

Борьба с избыточным потреблением токенов решается через лимиты на каждый запрос и refine-алгоритмы, которые суммаризируют промпты до нужного размера, уменьшая объем данных, отправляемых в LLM.

## Когда MCP стоит использовать, а когда — нет

:::columns Когда использовать MCP
### Множество источников данных
Интеграция AI с несколькими бизнес-приложениями и источниками данных, проведение математических расчетов — MCP сокращает количество интеграций.

### Разграничение доступов
Создание многофункциональных AI-ассистентов, где нужно строго разграничивать доступ к данным для разных LLM.

### Автоматизация DevOps
Стандартизация процессов разработки — шаблоны для REST API, автоматическая генерация приложений, деплой Helm-чартов для Kubernetes.

### Готовые MCP-серверы
Если провайдер предоставляет официальный MCP (например, для PostgreSQL), можно подключить его без разработки.
:::

Но у MCP есть ограничения. Это дополнительный уровень инфраструктуры, который нужно настраивать и поддерживать. Каждый MCP-сервер — это плюс один hop по сети и увеличение latency. Для простых задач с одним источником данных MCP может быть избыточным. Если внешние сервисы не поддерживают MCP или требуют серьезной адаптации, экономический эффект может не оправдать вложений. Наконец, безопасность полностью на разработчике — протокол ничего не несет «из коробки».

## Перспективы MCP: от хостинга до новой модели монетизации

Технология развивается в нескольких направлениях. MCP-серверы уходят в облако как SaaS-продукты, но одновременно растет запрос на on-premise решения. Одна из главных перспектив — добавление единого стандарта аутентификации и авторизации прямо в протокол, например, OAuth 2.1 «из коробки». Это позволит сразу получать готовый механизм безопасности при интеграции.

> Если разработчики будут делать все свои SaaS-продукты как MCP-серверы, а все веб-интерфейсы — как клиенты, готовые подключать MCP, то к любому приложению можно будет подключать новых агентов практически из коробки — просто заплатив вендору.

Стандартизация клиентского опыта — еще одна перспектива. Рынок AI-технологий меняется стремительно: от простых запросов к LLM — к агентам, мультиагентным системам, а теперь и к MCP. Протокол призван это стандартизировать, чтобы все было совместимо со всем.

## Итоги

Проект Syngenta x Raft показывает практическую ценность MCP-подхода для проектов с множеством разрозненных источников данных. Три MCP-сервера — антифрод, LLM-обращения к API и SQL-запросы — дают архитектуру, которую можно масштабировать без экспоненциального роста интеграций. При этом безопасность и корректность SQL-запросов остаются задачами, которые решаются постоянной доработкой промптов и валидацией, а не протоколом.

MCP — не серебряная пуля, а инженерный стандарт. Он экономит время и деньги там, где много источников данных и несколько LLM, но добавляет инфраструктурную сложность, которая не оправдана для простых проектов с одним агентом и одним источником.
