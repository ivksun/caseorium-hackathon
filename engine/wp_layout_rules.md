# правила верстки кейсов для WordPress

инструкция для AI-райтера и AI-редактора: как форматировать markdown, чтобы он корректно верстался в WordPress.

---

## заголовки и хлебные крошки

### 4 уровня заголовков в кейсе

1. **WordPress title (хлебная крошка)** = `заголовок_для_карточки_кейса`
   - короткое название: «[Компания] и [суть кейса]»
   - пример: «Т-Банк и AI-агент, вошедший в топ-3 операторов»
   - НЕ повторяет H1

2. **Заголовок H1** = полный заголовок статьи
   - формат: «Как [Компания] + [действие] + [результат с метрикой]»
   - пример: «Как Т-Банк научил AI-агента работать в интерфейсах операторов, войти в топ-3 по качеству и автоматизировать 45% обращений»

3. **Заголовок для карточки кейса** = хлебная крошка (дублирует п.1)

4. **Описание для карточки кейса**
   - одна фраза, начинается с СУЩЕСТВИТЕЛЬНОГО или ПРИЧАСТИЯ
   - НЕ начинается с глагола
   - пример: «Автоматизация 45% обращений с помощью computer-use агента»
   - НЕ: «Автоматизировал 45% обращений»

### в markdown

```
## Для дизайна:

Заголовок / подзаголовок на карточке:
[короткое название для хлебной крошки]

...

Дескрипшен: [существительное/причастие + ключевая метрика]
```

## описание компании (интро)

текст между `## [H1 заголовок]` и `## Для дизайна:`.

**ПРАВИЛО:** НИКОГДА не выделять жирным. без ** звездочек. plain text.

```
ПРАВИЛЬНО:
Т-Банк — один из крупнейших цифровых банков России с 53,7 млн клиентов.

НЕПРАВИЛЬНО:
Т-Банк — один из крупнейших цифровых банков России с **53,7 млн клиентов**.
```

---

## выделение жирным

**ПРАВИЛО:** жирным выделять ТОЛЬКО числа и метрики.

спроси себя: это конкретная цифра или метрика? если нет → не выделять.

```
ПРАВИЛЬНО:
- **45%** трафика автоматизировано
- **98 баллов** качества
- **T-Pro на 32 миллиарда параметров**

НЕПРАВИЛЬНО:
- **computer-use агентов** (это не метрика)
- **общие системы побеждают** (это не цифра)
- **скриптовый домен** (это описание)
```

## цитаты на плашках

блоки `> текст` превращаются в плашки (note_text) в WordPress.

**ПРАВИЛО:** текст на плашке должен быть литературным, не разговорным.

```
ПРАВИЛЬНО:
> Агент наследует ограничения, предусмотренные для операторов: если у сотрудника нет доступа к операции, у AI-агента его тоже нет.

НЕПРАВИЛЬНО:
> «Вообще минимально менять пришлось. В этом и есть смысл — мы используем все существующее»
```

убирать: «вот», «ну», «короче», «как бы», «типа».
не копировать устную речь дословно — переформулировать.

## пошаговые процессы (waypoint)

если в тексте есть ### Шаг первый / Шаг второй — они превращаются в waypoint блок с нумерацией.

**ПРАВИЛО:** НЕ дублировать «Шаг N:» в тексте. цифра уже стоит в заголовке блока.

**ПРАВИЛО:** текст шага — максимум 1-2 предложения. краткое описание, не подробный разбор.

```
ПРАВИЛЬНО:
### Записать траектории людей
Операторы работают как обычно, а система записывает логи действий в формате JSON.

НЕПРАВИЛЬНО:
### Шаг первый: записать траектории людей
Операторы работают как обычно и ничего не видят, а система записывает логи того, что они делают в интерфейсе. Данные сохраняются в формате JSON: состояние экрана в распарсенном виде и действие оператора. Таким образом формируется постоянно актуализируемый датасет — сотни тысяч строк каждый день.
```

---

## две проблемы/решения → две колонки

если в секции описываются 2 параллельных поинта (проблема + решение), форматируй их через `**Проблема 1:**` и `**Проблема 2:**`.

паблишер автоматически разнесет их в 2 колонки.

**ПРАВИЛО:** после описания проблем, общий текст (результаты, выводы) — через двойной перенос строки. он пойдет в отдельный блок.

```
**Проблема 1: информационные действия.** Оператор может ходить по разным разделам... Решение — агенты сами выполняют информационные действия.

**Проблема 2: долгое ожидание.** Пока агент выполняет действия, оператор ждет. Решение — до пяти агентов параллельно.

Параллельная работа потребовала изменения мотивации...
[этот текст пойдет в отдельный блок]
```

## картинки из слайдов

### какие слайды брать
- схемы архитектуры
- скриншоты интерфейсов / демо
- диаграммы процессов
- роадмапы

### какие НЕ брать
- слайды с только текстом
- слайды с только цифрами/метриками (покрываем текстом)
- абстрактные слайды без конкретики

### подписи
- максимум одна строка
- с большой буквы
- описание того, что на картинке

### две картинки подряд
→ автоматически становятся галереей/слайдером

### ПРАВИЛО: максимум 2 картинки подряд
НИКОГДА не ставить 3+ иллюстраций подряд без текста между ними. это выглядит как свалка.
если визуалов 3 — раздели текстом: картинка, абзац, картинка, абзац, картинка.

## блок метрик (ключевые результаты)

формат в markdown:

```
### Ключевые результаты проекта

**45%+**
трафика автоматизировано

**98 баллов**
качества по 100-балльной шкале
```

паблишер превращает это в карточки с зеленым цветом (#24DD63), по 3 в ряд.

**ПРАВИЛО:** ключевые результаты идут ПЕРЕД итогами (текстовым заключением).

---

## порядок блоков в конце кейса

1. Ключевые результаты проекта (карточки с цифрами)
2. Итоги (текстовое заключение)
3. CTA (автоматический)

---

## CTA

добавляется автоматически паблишером. всегда «Получить консультацию».
текст CTA контекстный, зависит от темы кейса.

---

## бегущая строка (tech stack)

добавляется ТОЛЬКО если в кейсе упоминаются 3+ конкретных инструментов/технологий.
только английские технические термины (T-Pro 32B, SFT, Kubernetes, RAGAS — ок).
НЕ общие понятия (GenAI, NLP, AI, LLM — слишком общо, не считаются).
если конкретных инструментов мало — блок не создается.

## TLDR

TLDR размечается как `### TLDR` и превращается в `note_text` блок с зеленой рамкой.

## rich-блоки: колонки, акценты, шаги, списки

WordPress поддерживает несколько ACF-блоков помимо простого текста.
используй markdown-разметку ниже — паблишер автоматически конвертирует в нужные блоки.

### карточки-колонки (layer_columns)

используй для: метрик, сравнений, параллельных идей (2-4 карточки в ряд).
заголовок карточки = зеленый, крупный (метрика или ключевая мысль).

markdown-формат:

```
:::columns Заголовок блока колонок (необязательный)
### С 32% до 41,5%
Выросла доля сотрудников, использующих AI

### 58%
Доля тех, кто использует AI иногда

### 26 проектов
Запущено после хакатона
:::
```

**ПРАВИЛА:**
- от 2 до 4 колонок
- заголовок колонки: метрика или короткая фраза (не абзац)
- текст колонки: 1-2 предложения
- НЕ использовать для длинных описаний — для этого есть обычный текст

### акцентный блок (layer_accent_text)

используй для: ключевых инсайтов, важных выводов, формул.
блок с фоном, выделяется визуально.

markdown-формат:

```
:::accent Заголовок акцентного блока (необязательный)
Ключевой инсайт: если люди это делают в интерфейсе — можно обучить агента повторять те же действия.
:::
```

**ПРАВИЛА:**
- максимум 2-3 предложения
- используй для самых важных мыслей, не для каждого абзаца
- максимум 2-3 акцентных блока на кейс

### нумерованные шаги (layer_waypoint)

используй для: пошаговых процессов, этапов, чек-листов.
каждый шаг получает номер автоматически.

markdown-формат:

```
:::steps
### Записать траектории людей
Операторы работают как обычно, а система записывает логи действий.

### Обучить модель
SFT-обучение на собранном датасете из действий операторов.

### Запустить агента
Агент начинает выполнять типовые операции автономно.
:::
```

**ПРАВИЛА:**
- НЕ дублировать номер в заголовке (числа ставятся автоматически)
- текст шага: 1-2 предложения максимум
- подходит для 3-6 шагов

### структурированный список (layer_list)

используй для: перечислений с заголовком и деталями (проекты-победители, департаменты, сценарии).

markdown-формат:

```
:::list Заголовок списка
### Команда «Умная полка»
- AI-ассистент, который помогает не забыть нужное в корзине
- Проект лег в основу развития клиентских сервисов

### Команда «Mindplex»
- Внутренний помощник для сотрудников
- Помогает быстро находить информацию в корпоративной базе знаний
:::
```

**ПРАВИЛА:**
- каждый ### = элемент списка с заголовком
- пункты через `- ` под каждым заголовком
- подходит для 2-6 элементов

### когда какой блок использовать

| ситуация | блок |
|---|---|
| 2-4 метрики / KPI | `:::columns` с зелеными заголовками |
| сравнение подходов / проблем | `:::columns` |
| ключевой инсайт / вывод | `:::accent` |
| пошаговый процесс | `:::steps` |
| перечисление проектов / сценариев с описанием | `:::list` |
| цитата / прямая речь спикера | `> текст` (note_text) |
| обычный текст | без разметки (custom_text) |

### сколько rich-блоков на кейс

рекомендация: **4-8 rich-блоков на кейс**.
- 1-2 блока `:::columns` (метрики, сравнения)
- 1 блок `:::steps` (если есть пошаговый процесс)
- 1-2 блока `:::accent` (ключевые инсайты)
- 1-2 блока `> цитата` (note_text)
- 0-1 блок `:::list` (если есть перечисления с описаниями)

НЕ перегружать — rich-блоки должны выделять, а не замусоривать. обычный текст остается основой.

## ЗАПРЕЩЕНО в тексте кейса

- ссылки на Telegram-каналы, соцсети, внешние ресурсы компании
- «подробнее в Telegram @...», «подписывайтесь», любые внешние CTA
- упоминание спикера по имени: НЕ «Алексей Носов называет», НЕ «по словам Носова»
- упоминание конференции/доклада как источника

правильно: «команда называет», «в компании объясняют», безличные конструкции.

## плотность текста

цель: максимально близко к тому, что говорит спикер. сохранить все детали, факты, технические подробности, примеры. убрать только разговорные обороты живой речи и явные повторы.

### главный принцип

текст кейса = структурированный пересказ выступления спикера. НЕ сокращенная выжимка. все важные детали, которые спикер озвучил, должны остаться в тексте.

### что убирать

- разговорные обороты: «вот», «ну», «как бы», «короче», «типа»
- явные повторы одной мысли в разных секциях (но если мысль в TLDR и в тексте — ок, TLDR это отдельный блок)
- нейросетевые вводные: «стоит отметить», «важно понимать», «в современном мире»

### что НЕ убирать

- технические детали (JSON формат, T-Pro 32B, SFT, доменная адаптация)
- конкретные примеры (кубышка vs кредитка, мышка в интерфейсе, премия Афанасию)
- объяснения ПОЧЕМУ что-то сделано (выбор домена, смена мотивации операторов)
- честность про ограничения и нерешенные проблемы
- метрики с контекстом (не просто «88%», а «88% точности на офлайн-метриках»)
- масштаб и контекст компании (20 000 процессов, десятки тысяч сотрудников)

### не описывай картинку подробно

если есть скриншот — не надо описывать каждый элемент интерфейса. одно-два предложения о сути: что тут происходит.

### структура

сколько секций нужно — столько и делай. не объединяй секции искусственно ради «компактности». если контроль и безопасность — отдельная тема, пусть будет отдельной секцией.
