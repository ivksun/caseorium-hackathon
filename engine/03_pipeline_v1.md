# pipeline кейсориум v1

автоматизированный процесс превращения расшифровки в готовый кейс

---

## A. вход

### что принимаем
→ расшифровка интервью/выступления со сцены (обязательно)  
→ презентация спикера (опционально)

### формат входных данных
- `.txt` / `.docx` / `.md` с текстом
- чистая расшифровка БЕЗ предобработки
- таймкоды можно оставить (игнорируются автоматически)

### что НЕ принимаем
✗ пересказы/саммари (нужна ПОЛНАЯ расшифровка)  
✗ статьи из медиа (только первоисточники)  
✗ маркетинговые тексты

---

## B. сборка черновика

### задача
собрать кейс по КАНОНИЧЕСКОЙ структуре с релевантным ToV, без додумывания фактов

### процесс

**шаг 1: извлечение фактов**
- проблема компании
- описание решения (шаги, процесс)
- метрики (цифры, результаты)
- технические детали
- инсайты спикера (прямые цитаты)

**шаг 2: поиск метаданных**
→ если в расшифровке НЕТ базовых данных о компании:
- название + позиция на рынке
- масштаб (клиенты, активы, сотрудники)
- топ-X в сегменте
→ искать в интернете (официальный сайт, отчеты, СМИ)

**шаг 3: структурирование**
применить каноническую структуру:

1. заголовок: `Как [Компания] + [действие] + [результат с метрикой]`
2. справка о компании
3. блок для дизайна (9 полей)
4. TLDR (проблема → решение → результат)
5. основные блоки:
   - контекст / проблема
   - решение (философия + шаги + технические детали)
   - результаты (разнотипные метрики)
   - масштабирование (если есть)
   - контроль
   - планы
   - итоги

**шаг 4: применение ToV**
→ предложения ЦЕЛЬНЫЕ (союзы, причастные обороты)  
→ подзаголовки ГОВОРЯЩИЕ (не "Решение", а "Пять шагов от траекторий до автономии")  
→ технические правила: ё→е, кавычки елочки, AI заглавными  
→ НЕТ клише ("это не про X, это про Y")  
→ НЕТ рекламных обещаний

### ограничения
✓ использовать ТОЛЬКО факты из расшифровки  
✓ если цифры нет — НЕ придумывать  
✓ если детали нет — оставить пробел / обозначить [уточнить]  
✓ прямые цитаты спикера — в кавычках елочках

### output
черновик кейса в `.md` с:
- полной структурой
- фактами из расшифровки
- метаданными (если нашли)
- пометками [уточнить] где данных не хватает

## C. валидация

### задача
проверить качество ФАКТОВ и логики, НЕ переписывать текст

### процесс

**1. проверка цифр**
☐ метрики совпадают с расшифровкой  
☐ нет противоречий внутри текста  
☐ единицы измерения указаны корректно  
☐ проценты/абсолютные значения не перепутаны

**2. проверка причинно-следственных связей**
☐ решение логично вытекает из проблемы  
☐ результаты связаны с действиями  
☐ масштабирование объясняет, как перешли от пилота к prod

**3. отделение фактов от интерпретаций**

факты (оставляем):
- "автоматизировали 45% трафика"
- "модель на 32 млрд параметров"
- "98 баллов по качеству"

интерпретации (маркируем):
- "это революционный подход" → [интерпретация, убрать]
- "лучшее решение на рынке" → [субъективная оценка, убрать]

**4. проверка структуры**
☐ все обязательные блоки на месте (TLDR, блок для дизайна, блок с цифрами)  
☐ последовательность логична  
☐ подзаголовки отражают суть блока

### output
**список проблем** (БЕЗ переписывания текста):

```
ЦИФРЫ:
- строка 45: проверить метрику "40% экономии" — в расшифровке неясно
- строка 89: добавить единицу измерения к "310 000"

ЛОГИКА:
- блок "Решение": пропущен переход от проблемы к выбору подхода
- блок "Результаты": не объяснено, как измерили качество

ФАКТЫ vs ИНТЕРПРЕТАЦИИ:
- строка 123: "революционный подход" → убрать, оставить факты
- строка 156: "лучшее решение" → субъективная оценка

СТРУКТУРА:
- отсутствует блок "Контроль"
- подзаголовок "Масштабирование" → сделать говорящим
```

## D. усиление

### задача
улучшить ЛОГИКУ и убрать воду, НЕ добавлять новые данные

### процесс

**1. улучшение логики**
→ усилить причинно-следственные связи  
→ добавить связки между блоками  
→ объяснить переходы (от проблемы к решению, от пилота к масштабированию)

пример:
```
было: "Компания решила использовать AI-агентов. Результат — 45% автоматизации."
стало: "Компания выбрала подход с AI-агентами, тк чат-боты закрывали только часть задач, 
а операторам нужен был инструмент для работы в сложных интерфейсах. Результат — 45% автоматизации."
```

**2. удаление воды**
→ общие фразы без смысла  
→ повторы одной и той же мысли  
→ клише  
→ длинные обороты (заменить на короткие)

убрать:
- "в современном мире"
- "как известно"
- "не секрет, что"
- "стоит отметить"
- "это не про X, это про Y"

**3. сжатие без потери смысла**
→ объединить предложения там, где логично  
→ убрать вводные конструкции  
→ заменить длинные обороты на короткие

пример:
```
было: "Стоит отметить, что компания приняла решение о том, что необходимо..."
стало: "Компания решила..."
```

### ограничения
✓ НЕ добавлять новые факты  
✓ НЕ менять цифры  
✓ НЕ додумывать технические детали  
✓ сохранить ВСЕ ключевые мысли из черновика

### output
усиленная версия кейса с:
- улучшенной логикой
- без воды
- сохраненными фактами

## E. финальный формат

### задача
применить ЕДИНЫЙ СТИЛЬ оформления для всех кейсов

### технические параметры

**шрифт:**
- arial для всего документа

**размеры:**
- 11 pt для обычного текста
- 16 pt для заголовков первого уровня (h1)
- 14 pt для заголовков второго уровня (h2)
- 11 pt для заголовков третьего уровня (h3), жирным

**отступы:**
- межстрочный интервал: 1,15
- отступы между абзацами: 6 pt после
- отступы между блоками: 12 pt после

**выделения:**
- жирным важные мысли по тексту (цифры, инсайты, технические детали)
- можно бегло читать только жирное и понять суть

### плашки (2-3 на кейс)

выделить 2-3 ключевые мысли на отдельные плашки:

**критерии отбора:**
- сильный инсайт (не очевидный)
- применимость для других компаний
- философия подхода
- честность про ограничения

**формат плашки:**
- 1-2 предложения
- можно с цитатой спикера
- визуально отделена от основного текста

пример:
```
┌─────────────────────────────────────────────────────────────┐
│ «Наши сотрудники так похожи на людей, что ничем не          │
│ отличаются во всех системах, даже пытались выплатить        │
│ премию Афанасию, начали искать, кому оплатить, и не нашли» │
└─────────────────────────────────────────────────────────────┘
```

### блок с цифрами

в разделе "Итоги" обязательно:

```
### Ключевые результаты проекта

**45%**  
трафика автоматизировано полностью автономно

**40%**  
экономия времени дополнительно к чат-боту

**98 баллов**  
получает AI-агент по 100-балльной системе
```

**формат:**
- цифра жирным
- описание с новой строки
- 5-7 ключевых метрик

### финальный чек-лист

**структура:**
☐ TLDR (проблема → решение → результат)  
☐ блок для дизайна (9 полей)  
☐ подзаголовки говорящие  
☐ блок с цифрами в итогах  
☐ вывод короткий (2 абзаца)  
☐ 2-3 плашки с инсайтами

**текст:**
☐ предложения цельные  
☐ жирным ключевые мысли  
☐ списки где уместно  
☐ нет разделителей ---  
☐ ё заменено на е  
☐ кавычки елочки «»  
☐ AI большими буквами

**содержание:**
☐ конкретная метрика в заголовке  
☐ легитимация через масштаб  
☐ проблема типовая для сегмента  
☐ решение = процесс (шаги)  
☐ минимум 3 конкретных сценария  
☐ разнотипные метрики (не только roi)  
☐ есть честность про несовершенство  
☐ метрики переведены в понятные единицы  
☐ нет додумывания фактов

### output
готовый кейс в `.docx` с применением:
- технических параметров (шрифт, размеры, отступы)
- выделений жирным
- плашек с инсайтами
- блока с цифрами
- единого стиля

## схема pipeline

```
ВХОД
  ↓
расшифровка + презентация (опционально)
  ↓
┌────────────────────────────────────────┐
│ B. СБОРКА ЧЕРНОВИКА                    │
│ → извлечение фактов                    │
│ → поиск метаданных                     │
│ → структурирование (канон)             │
│ → применение ToV                       │
│ БЕЗ додумывания                        │
└────────────────────────────────────────┘
  ↓
черновик.md
  ↓
┌────────────────────────────────────────┐
│ C. ВАЛИДАЦИЯ                           │
│ → проверка цифр                        │
│ → проверка логики                      │
│ → факты vs интерпретации               │
│ → структура                            │
│ БЕЗ переписывания текста               │
└────────────────────────────────────────┘
  ↓
список проблем
  ↓
┌────────────────────────────────────────┐
│ D. УСИЛЕНИЕ                            │
│ → улучшение логики                     │
│ → удаление воды                        │
│ → сжатие без потери смысла             │
│ БЕЗ добавления новых данных            │
└────────────────────────────────────────┘
  ↓
усиленная версия.md
  ↓
┌────────────────────────────────────────┐
│ E. ФИНАЛЬНЫЙ ФОРМАТ                    │
│ → применение технических параметров    │
│ → выделение жирным                     │
│ → плашки (2-3)                         │
│ → блок с цифрами                       │
│ → единый стиль                         │
└────────────────────────────────────────┘
  ↓
ГОТОВЫЙ КЕЙС.docx
```

## время выполнения (оценочно)

**B. сборка черновика:** 30-40 мин  
**C. валидация:** 10-15 мин  
**D. усиление:** 15-20 мин  
**E. финальный формат:** 10 мин

**итого:** ~60-85 мин на кейс

---

## критичные принципы

1. **НЕ додумывать факты** — только то, что в расшифровке
2. **валидация ≠ переписывание** — список проблем, не правки текста
3. **усиление ≠ добавление** — убирать воду, не добавлять данные
4. **единый стиль** — все кейсы выглядят одинаково (шрифт, отступы, плашки)
