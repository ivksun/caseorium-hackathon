# деконструкция кейса совкомбанк

## основная мысль (не поверхностная)

НЕ "смотрите, какой крутой бот мы сделали"  
А "ai решает РЕАЛЬНУЮ боль крупных компаний — недоступность знаний и перегрузку команд"

кейс доказывает: genai — это не hype, а инструмент для измеримой экономии ресурсов

---

## архитектура убеждения: как ведёт читателя

### 1. ЗАГОЛОВОК → конкретика вместо обещаний
"310 000 рабочих часов в год" — не "повысили эффективность", а КОНКРЕТНАЯ цифра  
→ создаёт якорь доверия с первой секунды

**принцип:** метрика в заголовке = проверяемое утверждение, не маркетинговый слоган

### 2. СПРАВКА О КОМПАНИИ → легитимация
"топ-10 по активам" + "36 000 сотрудников"  
→ это не стартап-эксперимент, это enterprise-решение

**принцип:** если клиент крупный — подчеркни масштаб. это снимает возражение "у них маленькая задача, нам не подойдёт"

### 3. КОНТЕКСТ → масштаб → боль
"36 000 человек" → "растёт штат" → "сложность процессов"  
→ от факта к последствию

**принцип:** не начинай с проблемы в лоб. покажи МАСШТАБ, чтобы проблема стала очевидной

### 4. ВЫЗОВ → формулировка проблемы как ОБЩЕЙ для категории
"привычные для крупных компаний вопросы" — не "у совкомбанка была проблема", а "У ВСЕХ КРУПНЫХ такое"  
→ читатель узнаёт себя

**принцип:** проблему формулируй как типовую для сегмента, а не уникальную для клиента

### 5. РЕШЕНИЕ → путь, а не магия
"масштабный аудит" → "создали сценарии" → "собрали базу знаний" → "связали с сервисами"  
→ показывает, что это РАБОТА, а не волшебная кнопка

**принцип:** описывай процесс, а не только результат. это создаёт ощущение контроля ("мы тоже так сможем")

### 6. КОНКРЕТИКА ИСПОЛЬЗОВАНИЯ → не функции, а задачи
НЕ "умеет искать документы"  
А "рассчитывает отпускные, больничные, командировочные"

**принцип:** не перечисляй возможности технологии, покажи РАБОЧИЕ СЦЕНАРИИ, где она применяется

## функции блоков (зачем каждый нужен)

| блок | функция | как работает |
|------|---------|--------------|
| **заголовок** | зацепить + дать якорь доверия | конкретная метрика вместо общих слов |
| **справка о компании** | легитимация | "если они смогли, значит это работает" |
| **контекст** | создать ощущение масштаба | от цифр к последствиям |
| **вызов** | формулировка проблемы | читатель узнаёт СВОЮ боль |
| **решение** | показать путь | не магия, а процесс (воспроизводимо) |
| **результаты** | доказать эффект | метрики разного типа (не только roi) |
| **планы** | показать живость проекта | это не "сделали и забыли", а развивается |
| **итоги** | социальное доказательство | премия hr impact = внешнее признание |
| **cta** | конверсия | привязан к кейсу, а не общий призыв |

## где убеждает vs где информирует

### УБЕЖДАЕТ:
- "привычные для крупных компаний вопросы" → ты не один такой
- "99% типовых вопросов закрывает бот" → почти всё автоматизировано
- "310 000 часов = 155 сотрудников" → перевод в понятную метрику
- "точность 72% → 89%, цель 97%" → честность + траектория роста
- "победитель премии hr impact" → внешнее признание

### ИНФОРМИРУЕТ:
- список задач, которые решает сова (отпускные, больничные и тд)
- технический стек ("комбинация моделей")
- цифры по запросам (90 000 запросов)

**принцип:** информация нужна для конкретики, убеждение — для снятия сомнений

## за счёт чего кейс "живой", а не формальный

1. **ЧЕСТНОСТЬ ПРО НЕСОВЕРШЕНСТВО**  
   "точность 72% → 89%, цель 97%"  
   → не скрывают, что не идеально. это создаёт доверие

2. **ЭВОЛЮЦИЯ, А НЕ МОМЕНТАЛЬНЫЙ УСПЕХ**  
   "первые месяцы", "через пять месяцев", "в планах"  
   → показывает развитие, а не "сделали за неделю"

3. **КОНКРЕТНЫЕ СЦЕНАРИИ ИСПОЛЬЗОВАНИЯ**  
   не "помогает hr", а "рассчитывает софинансирование в зависимости от стажа"  
   → детали делают кейс осязаемым

4. **ПЕРЕВОД МЕТРИК В ПОНЯТНЫЕ ЕДИНИЦЫ**  
   "310 000 часов = 155 сотрудников"  
   → абстрактное становится конкретным

5. **БЛОК "ЧТО ДАЛЬШЕ"**  
   → проект живёт, развивается, это не одноразовая история

**принцип:** живость = честность + конкретика + динамика развития

## элементы доверия

### 1. СОЦИАЛЬНОЕ ДОКАЗАТЕЛЬСТВО
- топ-10 банков → авторитет компании
- премия hr impact → внешнее признание
- 36 000 сотрудников → масштаб внедрения

### 2. ПРОВЕРЯЕМЫЕ МЕТРИКИ
- 310 000 часов в год
- 99% типовых вопросов
- 90 000 запросов
- 80%+ довольных пользователей
→ конкретика, а не "повысили эффективность"

### 3. ЧЕСТНОСТЬ ПРО ОГРАНИЧЕНИЯ
- "точность 72% в начале" (не скрывают несовершенство)
- "цель 97%" (признают, что ещё есть путь)
→ это не приукрашивание, а реальная история

### 4. ТЕХНИЧЕСКИЕ ДЕТАЛИ
- "защищенный контур банка"
- "комбинация моделей + собственная модель"
- "модули для русскоязычных терминов"
→ показывает глубину проработки

### 5. ПРОЦЕСС, А НЕ МАГИЯ
- "масштабный аудит"
- "изучили запросы"
- "создали сценарии"
→ это РАБОТА, а не волшебство

**принцип:** доверие = авторитет + конкретика + честность + процесс

## воспроизводимые принципы для других кейсов

### структура нарратива
1. **конкретная метрика в заголовке** (не "повысили", а "310 000 часов")
2. **легитимация через масштаб** (топ-10, 36к сотрудников)
3. **от контекста к боли** (не сразу "была проблема", а "вот масштаб → из-за этого проблема")
4. **проблема как типовая для категории** ("привычные для крупных компаний")
5. **решение как процесс** (аудит → сценарии → база знаний → интеграция)
6. **конкретные рабочие сценарии** (не "ищет документы", а "рассчитывает отпускные")
7. **честность про несовершенство** (72% → 89% → цель 97%)
8. **перевод метрик в понятные единицы** (310к часов = 155 сотрудников)
9. **социальное доказательство** (премия, признание)
10. **блок "что дальше"** (проект живёт и развивается)

### где убеждать
- масштаб компании (если enterprise)
- типовость проблемы для сегмента
- честность про ограничения
- траектория роста метрик
- внешнее признание

### где информировать
- конкретные задачи/сценарии использования
- технические детали (если релевантны для доверия)
- процесс создания
- цифры (запросы, пользователи)

### что создаёт доверие
- проверяемые метрики
- честность про недостатки
- процесс вместо магии
- социальное доказательство
- детали реализации

### что делает кейс живым
- эволюция (было → стало → будет)
- несовершенство в начале
- планы развития
- конкретика сценариев (не функции, а задачи)
- перевод абстракций в конкретику

---

## КРИТИЧЕСКИ ВАЖНО

кейс работает НЕ потому что "хорошо написан"  
а потому что **структура нарратива снимает возражения по очереди:**

1. "это вообще работает?" → метрика в заголовке
2. "это для маленьких компаний?" → топ-10 банк, 36к сотрудников
3. "у нас другая проблема?" → "привычные для крупных компаний"
4. "это сложно внедрить?" → показан процесс (аудит → сценарии → база)
5. "это только теория?" → 90 000 запросов, конкретные сценарии
6. "это идеально сразу?" → нет, 72% → 89%, цель 97%
7. "это признают другие?" → премия hr impact

**воспроизводимый принцип:** структура кейса = последовательное закрытие возражений через конкретику, честность и социальное доказательство

## кейс как история для рационального читателя

### где возникает напряжение и интерес

#### 1. ЗАГОЛОВОК: ЦИФРОВАЯ ПРОВОКАЦИЯ
"310 000 часов в год"  
→ мозг рационального читателя немедленно начинает считать: "сколько это людей? сколько денег?"  
→ **напряжение:** я не знаю, как они это сделали, но хочу узнать

**принцип:** конкретная огромная цифра = когнитивный зуд, который нужно почесать

#### 2. КОНТЕКСТ: НАРАСТАНИЕ СЛОЖНОСТИ
"36 000 человек" → "масштаб влияет на сложность" → "новые продукты" → "растёт штат"  
→ читатель чувствует **эскалацию**: чем дальше, тем сложнее  
→ появляется вопрос: "как они вообще с этим справляются?"

**принцип:** покажи нарастание проблемы, а не статичное состояние

#### 3. ВЫЗОВ: УЗНАВАНИЕ СЕБЯ
"привычные для крупных компаний вопросы"  
→ **напряжение:** это про меня, у меня та же боль  
→ читатель уже вовлечён эмоционально (да, рациональный читатель тоже реагирует на узнавание)

**принцип:** если читатель узнаёт свою проблему — он уже не читатель, он участник истории

#### 4. РЕШЕНИЕ: РАЗРЯДКА ЧЕРЕЗ ПРОЦЕСС
"масштабный аудит → сценарии → база знаний"  
→ **разрядка напряжения:** ага, значит это сделали методично, не волшебством  
→ но тут же **новое напряжение:** "а как это работает в реальности?"

**принцип:** разрядка одного напряжения → создание следующего вопроса

#### 5. КОНКРЕТИКА ИСПОЛЬЗОВАНИЯ: ОСЯЗАЕМОСТЬ
"рассчитывает отпускные, больничные, командировочные"  
→ **разрядка:** теперь я вижу, КАК это работает  
→ читатель может представить это в своём контексте

**принцип:** детализация сценариев = переход от абстракции к визуализации

#### 6. МЕТРИКИ: ВТОРАЯ ВОЛНА ИНТЕРЕСА
"99% типовых вопросов", "310 000 часов = 155 сотрудников"  
→ **новое напряжение:** это правда такие цифры?  
→ читатель проверяет логику, считает

**принцип:** метрики в середине кейса = повторная зацепка для скептиков

#### 7. ЧЕСТНОСТЬ: НЕОЖИДАННЫЙ ПОВОРОТ
"точность 72% → 89%, цель 97%"  
→ **разрядка скептицизма:** они не врут, не всё идеально  
→ **новый интерес:** если они честны в этом, значит остальное тоже правда

**принцип:** признание несовершенства = усиление доверия к совершенным метрикам

#### 8. ЧТО ДАЛЬШЕ: ОТКРЫТЫЙ ФИНАЛ
"интеграция с внешними сервисами", "применение в других компаниях экосистемы"  
→ **незакрытое напряжение:** история не закончена, проект живой  
→ читатель остаётся в контакте с темой

**принцип:** открытый финал > закрытый "мы победили". живая история притягивает

### какие вопросы читателя предугадываются и закрываются

| момент в кейсе | вопрос читателя | как закрывается |
|----------------|-----------------|-----------------|
| **заголовок:** 310к часов | "это реально или маркетинг?" | конкретная цифра + название крупного банка |
| **справка:** топ-10 банк | "у них маленькая задача?" | 36 000 сотрудников = enterprise-масштаб |
| **контекст:** растёт сложность | "почему им это вообще нужно?" | чем больше людей → тем сложнее процессы |
| **вызов:** недоступность знаний | "у нас другая проблема?" | "привычные для крупных компаний" = типовая боль |
| **решение:** аудит + сценарии | "как они это сделали?" | показан процесс шаг за шагом |
| **интеграция:** портал, мобилка, мессенджер | "а где это работает?" | везде, где работают сотрудники |
| **сценарии:** отпускные, больничные | "что конкретно умеет?" | не абстракция, а реальные задачи |
| **технологии:** комбинация моделей | "это безопасно для банка?" | "защищённый контур", собственная модель |
| **метрики:** 99%, 80%+ довольны | "это сработало?" | конкретные цифры использования |
| **честность:** 72% → 89% | "это идеально?" | нет, есть путь роста → честность |
| **признание:** премия hr impact | "это только они так считают?" | внешнее признание = объективность |
| **планы:** интеграция, масштабирование | "это разовая история?" | нет, проект развивается |

**принцип:** хороший кейс = предугаданные возражения, закрытые ДО того, как читатель успел их сформулировать

### где читателя держат, где могли потерять

#### ДЕРЖАТ:

**1. КОНКРЕТИКА ВМЕСТО АБСТРАКЦИЙ**  
"рассчитывает софинансирование по соцпрограммам в зависимости от стажа"  
→ детали делают историю РЕАЛЬНОЙ, а не теоретической

**2. ПЕРЕВОД МЕТРИК В ПОНЯТНЫЕ ЕДИНИЦЫ**  
"310 000 часов = 155 сотрудников"  
→ абстрактное число становится ОСЯЗАЕМЫМ

**3. РИТМ: ПРОБЛЕМА → РЕШЕНИЕ → ДОКАЗАТЕЛЬСТВО**  
чёткая структура, читатель знает, что идёт дальше

**4. КОРОТКИЕ БЛОКИ**  
нет простыней текста, всё разбито на секции

**5. ЧЕСТНОСТЬ**  
"72% → 89%" → читатель расслабляется, ему не врут

#### МОГЛИ ПОТЕРЯТЬ:

**1. ТЕХНИЧЕСКИЙ БЛОК: "комбинация моделей"**  
→ **риск:** слишком абстрактно для нетехнических читателей  
→ **спасает:** следом идёт "защищённый контур" (понятная боль банка)

**2. ПЕРЕЧИСЛЕНИЕ ЗАДАЧ: "отпускные, больничные, командировочные"**  
→ **риск:** список = скучно  
→ **спасает:** конкретность (не "помогает hr", а "рассчитывает")

**3. БЛОК "ЧТО ДАЛЬШЕ"**  
→ **риск:** планы = не факты, могут показаться пустыми обещаниями  
→ **спасает:** он идёт ПОСЛЕ метрик, когда доверие уже есть

**4. ОТСУТСТВИЕ ДРАМЫ**  
→ **риск:** нет истории про "как было плохо"  
→ **спасает:** масштаб (36к человек) создаёт ощущение сложности без драматизации

**принцип:** держат детали + ритм + честность. теряют абстракции + списки + обещания без доказательств

### за счёт чего легко читается (несмотря на сложную тему)

#### 1. СТРУКТУРА КАК ЛЕСТНИЦА
каждый блок отвечает на ОДИН вопрос:
- кто? → совкомбанк, топ-10
- зачем? → 36к человек, сложность процессов
- что за проблема? → недоступность знаний
- как решили? → аудит → сценарии → база
- что получилось? → 310к часов, 99% вопросов
- это правда? → 72% → 89% (честность)
- признают другие? → премия

**принцип:** один блок = один вопрос. не смешивай

#### 2. ОТСУТСТВИЕ ЖАРГОНА (или объяснение)
НЕТ: "мы применили rag-архитектуру с векторной индексацией"  
ЕСТЬ: "база знаний", "защищённый контур"

**принцип:** если термин не общеупотребим — не используй или объясни сразу

#### 3. КОНКРЕТИКА ВМЕСТО ПРИЛАГАТЕЛЬНЫХ
НЕ "эффективный помощник"  
А "поиск занимает 20-40 секунд вместо нескольких минут"

**принцип:** убери все прилагательные, замени на измеримые факты

#### 4. ВИЗУАЛИЗАЦИЯ ЧЕРЕЗ СЦЕНАРИИ
"рассчитывает отпускные" → читатель видит картинку: человек задаёт вопрос, получает расчёт  
→ мозг обрабатывает истории легче, чем абстракции

**принцип:** описывай не возможности, а ДЕЙСТВИЯ в контексте

#### 5. ЦИФРЫ В КОНТЕКСТЕ
"90 000 запросов" + "первые месяцы"  
→ не просто цифра, а цифра во времени

**принцип:** каждая метрика должна быть привязана к контексту (время, масштаб, сравнение)

#### 6. КОРОТКИЕ ПРЕДЛОЖЕНИЯ В КЛЮЧЕВЫХ МЕСТАХ
"его цель — сделать получение ответов максимально быстрым"  
→ простая конструкция, легко схватывается

**принцип:** сложные мысли = короткие предложения

---

### приёмы против сухости (не превращая в инфоцыганство)

#### 1. ПЕРСОНИФИКАЦИЯ РЕШЕНИЯ
"сова" вместо "чат-бот версия 2.1"  
→ название создаёт образ, но не выглядит легкомысленно (сова = мудрость)

**принцип:** если можно дать решению имя — дай. но не "супербот 3000"

#### 2. АКТИВНЫЙ ЗАЛОГ
НЕ "была проведена интеграция"  
А "команда связала чат-бот с внутренними сервисами"

**принцип:** активный залог = видна живая работа людей

#### 3. ТРАЕКТОРИЯ ВМЕСТО СТАТИКИ
"72% → 89% → цель 97%"  
→ это ДВИЖЕНИЕ, а не застывший результат

**принцип:** покажи процесс изменения, а не точку "после"

#### 4. ОТКРЫТЫЙ ФИНАЛ
"в планах — интеграция", "рассматривают применение в других компаниях"  
→ история не закончена = она живая

**принцип:** не заканчивай на "мы победили". покажи, что проект развивается

#### 5. ЧЕСТНОСТЬ КАК ДРАМАТУРГИЯ
"точность 72% в начале"  
→ несовершенство = человечность, без потери серьёзности

**принцип:** одно признание недостатка делает все достижения правдоподобнее

#### 6. ПЕРЕВОД АБСТРАКЦИЙ В ВИЗУАЛЬНОЕ
"310 000 часов = 155 сотрудников"  
→ читатель видит офис со 155 людьми

**принцип:** к каждой абстрактной метрике добавь материальную метафору

#### 7. СОЦИАЛЬНОЕ ДОКАЗАТЕЛЬСТВО В ФИНАЛЕ
"премия hr impact"  
→ не банк сам себя хвалит, а внешнее признание

**принцип:** завершай не своей оценкой, а оценкой третьих лиц

## воспроизводимые приёмы для удержания рационального читателя

### создание напряжения:
1. **провокационная цифра в заголовке** (заставляет считать)
2. **нарастание сложности** (масштаб → последствия)
3. **узнавание себя** ("привычные для крупных компаний")
4. **открытые вопросы** (показал проблему → закрыл через N абзацев)

### закрытие вопросов:
1. **предугадывай возражения** и закрывай их ДО того, как читатель успел сформулировать
2. **каждый блок отвечает на один вопрос** (не смешивай)
3. **доказательства разного типа** (метрики + процесс + соцдоказательство)

### против потери внимания:
1. **короткие блоки** (один вопрос = один блок)
2. **конкретика вместо абстракций** (не "эффективно", а "20-40 секунд")
3. **ритм:** проблема → решение → доказательство
4. **честность** (один недостаток делает всё остальное правдоподобным)

### против сухости:
1. **персонификация** ("сова" вместо "система 2.1")
2. **активный залог** (команда сделала, а не было сделано)
3. **траектория** (72% → 89%, а не просто 89%)
4. **открытый финал** (история продолжается)
5. **визуализация через сценарии** (описывай действия, а не возможности)
6. **перевод метрик** (310к часов = 155 человек)
7. **социальное доказательство** (премия = не мы себя хвалим)

кейс для рационального читателя — это НЕ отсутствие эмоций  
это **контролируемое напряжение через когнитивные зацепки:**

- огромная цифра → хочу понять, как
- узнавание проблемы → это про меня
- процесс → я могу воспроизвести

- соцдоказательство → не только они так думают

# кейс как операционная система

## входные ожидания читателя (что он хочет получить)

### по типу читателя:

#### 1. C-LEVEL / ЛПР
**входит с вопросами:**
- это работает на практике или hype?
- сколько это стоит в ресурсах?
- какие риски?
- кто ещё так делает?

**ожидает:**
- конкретные метрики roi
- масштаб компании-клиента (сравнимый с его)
- социальное доказательство (кто ещё признаёт это)
- честность про ограничения

**уходит с решением:** инвестировать или нет

#### 2. ПРОДУКТОВИКИ / СТАРТАПЕРЫ
**входит с вопросами:**
- какие сценарии реально работают?
- что можно украсть/адаптировать?
- какой стек использовали?

**ожидает:**
- конкретные use case
- процесс создания
- технические детали (но не перегруз)

**уходит с идеями:** что применить у себя

#### 3. ИНЖЕНЕРЫ / ИССЛЕДОВАТЕЛИ
**входит с вопросами:**
- как это технически устроено?
- какие модели?
- какие проблемы решали?

**ожидает:**
- технические детали
- честность про ограничения
- траектория улучшений (72% → 89%)

**уходит с пониманием:** как воспроизвести

#### 4. КОНСУЛЬТАНТЫ / АНАЛИТИКИ
**входит с вопросами:**
- можно ли это использовать как аргумент?
- насколько это проверяемо?
- есть ли внешнее признание?

**ожидает:**
- проверяемые метрики
- социальное доказательство
- типизацию проблемы (не уникальная, а общая для сегмента)

**уходит с материалом:** для клиентов/отчётов

### общее для всех типов:

✓ **конкретика** (не "эффективно", а цифры)  
✓ **честность** (не идеал, а реальность)  
✓ **применимость** (могу ли я это использовать)  
✓ **доверие** (это правда или маркетинг)

## ключевые точки внимания (где останавливается взгляд/мозг)

### 1. ЗАГОЛОВОК [0 сек]
**"310 000 рабочих часов в год"**

→ **реакция мозга:** это сколько? считаю → 155 человек  
→ **решение:** читать или пропустить

**критично:** конкретная метрика, которая заставляет СЧИТАТЬ

### 2. СПРАВКА О КОМПАНИИ [5-10 сек]
**"топ-10 по активам, 36 000 сотрудников"**

→ **реакция мозга:** это мой масштаб или нет?  
→ **решение:** релевантно мне или нет

**критично:** легитимация через масштаб/авторитет

### 3. ФОРМУЛИРОВКА ПРОБЛЕМЫ [20-30 сек]
**"привычные для крупных компаний вопросы"**

→ **реакция мозга:** узнавание ("это про меня")  
→ **эмоция:** вовлечённость

**критично:** типизация проблемы (не уникальная, а общая)

### 4. ПРОЦЕСС РЕШЕНИЯ [40-60 сек]
**"масштабный аудит → сценарии → база знаний"**

→ **реакция мозга:** это воспроизводимо или магия?  
→ **оценка:** могу ли я повторить

**критично:** показан путь, а не только результат

### 5. КОНКРЕТНЫЕ СЦЕНАРИИ [60-90 сек]
**"рассчитывает отпускные, больничные, командировочные"**

→ **реакция мозга:** визуализация (вижу, как это работает)  
→ **эмоция:** "я могу представить это у себя"

**критично:** задачи, а не функции технологии

### 6. МЕТРИКИ [90-120 сек]
**"99% типовых вопросов, 310к часов = 155 сотрудников"**

→ **реакция мозга:** проверка (логично ли это?)  
→ **оценка:** верю или нет

**критично:** разные типы метрик + перевод в понятные единицы

### 7. ЧЕСТНОСТЬ ПРО НЕСОВЕРШЕНСТВО [120-140 сек]
**"точность 72% → 89%, цель 97%"**

→ **реакция мозга:** неожиданность (они не врут)  
→ **эмоция:** доверие резко растёт

**критично:** ОДНО признание недостатка (не больше)

### 8. СОЦИАЛЬНОЕ ДОКАЗАТЕЛЬСТВО [140-160 сек]
**"победитель премии hr impact"**

→ **реакция мозга:** не только они так считают  
→ **решение:** использовать этот кейс как ориентир

**критично:** внешнее признание (не самовосхваление)

### 9. ОТКРЫТЫЙ ФИНАЛ [160-180 сек]
**"в планах — интеграция, масштабирование"**

→ **реакция мозга:** это живой проект  
→ **ощущение:** история продолжается

**гибко:** можно не включать, но усиливает эффект

### 10. CTA [180+ сек]
**"хотите внедрить ai-помощника?"**

→ **реакция мозга:** что мне делать с этим знанием?  
→ **действие:** оставить заявку или запомнить бренд

**критично:** привязан к конкретному кейсу, а не общий призыв

## логика раскрытия решения (последовательность информации)

### ФАЗА 1: ЛЕГИТИМАЦИЯ [0-30 сек]
**задача:** установить авторитет + релевантность

```
заголовок (метрика) 
  ↓
справка о компании (масштаб)
  ↓
контекст (почему им это нужно)
  ↓
формулировка проблемы (узнавание)
```

**критично:** эта последовательность. если начать с проблемы → нет контекста масштаба

### ФАЗА 2: РЕШЕНИЕ КАК ПРОЦЕСС [30-90 сек]
**задача:** показать воспроизводимость

```
что сделали (аудит → сценарии → база)
  ↓
где это работает (портал, мобилка, мессенджер)
  ↓
конкретные задачи (отпускные, больничные)
  ↓
технические детали (защищённый контур, модели)
```

**критично:** от общего к частному. сначала процесс, потом детали

### ФАЗА 3: ДОКАЗАТЕЛЬСТВО [90-140 сек]
**задача:** закрыть скептицизм

```
метрики результата (99%, 310к часов)
  ↓
перевод в понятное (155 сотрудников)
  ↓
честность про несовершенство (72% → 89%)
  ↓
планы развития (интеграция, масштабирование)
```

**критично:** честность идёт ПОСЛЕ больших метрик, а не до

### ФАЗА 4: СОЦИАЛЬНОЕ ПРИЗНАНИЕ [140-180 сек]
**задача:** перевести из "они так считают" в "все так считают"

```
итоги (изменило коммуникации)
  ↓
внешнее признание (премия)
  ↓
обобщение (цифровая трансформация = конкретные действия)
```

**критично:** внешнее признание в конце, когда уже есть доверие

### ФАЗА 5: КОНВЕРСИЯ [180+ сек]
**задача:** дать действие

```
cta привязанный к кейсу
  ↓
краткое описание продукта
  ↓
пример ("как это сделал совкомбанк")
```

**гибко:** формулировка, но не отсутствие

"топ-10", "36 000 сотрудников"

снимает возражение "это игрушечная задача"

#### 5. РЕШЕНИЕ КАК ПРОЦЕСС
аудит → сценарии → база → интеграция

"рассчитывает отпускные", а не "помогает hr"

#### 7. ЧЕСТНОСТЬ ПРО НЕСОВЕРШЕНСТВО (одно признание)

премия / внешнее признание

**почему критично:**  
переводит из "они так считают" в "все так считают"

не только roi, но и удовлетворённость, скорость, охват

3-5 задач vs 7-10 задач

**почему гибко:**  
главное — не меньше 3 (для доказательства системности)

#### 6. ФОРМУЛИРОВКА CTA
"хотите внедрить?" vs "узнайте больше"

**почему гибко:**  
главное — привязан к кейсу, а не общий "свяжитесь с нами"

#### 7. ПОРЯДОК МЕТРИК ВНУТРИ БЛОКА
можно начать с roi или с удовлетворённости

**почему гибко:**  
главное — наличие разнотипных метрик, порядок вторичен

#### 8. СТЕПЕНЬ ДЕТАЛИЗАЦИИ ПРОЦЕССА
"аудит → сценарии → база" vs развёрнутое описание каждого шага

**почему гибко:**  
для инженеров нужна детализация, для c-level — краткость

#### 9. ЭМОЦИОНАЛЬНАЯ ОКРАСКА
можно более сдержанно или более живо

**почему гибко:**  
главное — активный залог + конкретика, тональность вторична

#### 10. ВИЗУАЛЬНЫЕ ЭЛЕМЕНТЫ
скриншоты, графики, иконки

**почему гибко:**  
усиливают восприятие, но текстовая структура работает и без них

## формула эталонного кейса (воспроизводимая система)

### ОБЯЗАТЕЛЬНЫЕ КОМПОНЕНТЫ (ядро системы):

```
1. заголовок = конкретная метрика
2. легитимация = масштаб/авторитет компании
3. контекст = почему проблема возникла
4. проблема = типовая для сегмента
5. решение = процесс (шаги), а не результат
6. сценарии = 3+ конкретных задач
7. метрики = разнотипные (roi + скорость + удовлетворённость)
8. перевод метрик = в понятные единицы
9. честность = одно признание несовершенства
10. социальное доказательство = внешнее признание
11. cta = привязанный к кейсу
```

### ПОСЛЕДОВАТЕЛЬНОСТЬ (критична):

```
легитимация (кто + масштаб)
  ↓
контекст → проблема (узнавание)
  ↓
решение (процесс + сценарии + техдетали)
  ↓
метрики (разнотипные + перевод)
  ↓
честность (одно признание)
  ↓
социальное доказательство
  ↓
cta
```

### ГИБКИЕ НАДСТРОЙКИ:

- персонификация решения
- степень технических деталей
- длина блоков
- блок "что дальше"
- визуальные элементы
- тональность (сдержанная ↔ живая)

## ОПЕРАЦИОНАЛЬНЫЙ ЧЕКЛИСТ ДЛЯ ГЕНЕРАЦИИ КЕЙСА

### перед написанием:

☐ определи тип целевого читателя (c-level / продуктовик / инженер / консультант)  
☐ найди ОДНУ главную метрику для заголовка  
☐ проверь масштаб компании (если enterprise — подчеркни, если нет — найди другую легитимацию)  
☐ сформулируй проблему как ТИПОВУЮ для сегмента  
☐ собери минимум 3 конкретных рабочих сценария  
☐ собери разнотипные метрики (не только roi)  
☐ найди ОДНО честное признание несовершенства  
☐ найди социальное доказательство (премия / признание / кто ещё так делает)

### структура (обязательный порядок):

1. **заголовок:** [компания] + [конкретная метрика]  
2. **справка:** масштаб/авторитет компании  
3. **контекст:** почему проблема возникла (от масштаба к последствиям)  
4. **вызов:** формулировка проблемы как типовой  
5. **решение:** процесс создания (аудит → сценарии → база → интеграция)  
6. **где работает:** каналы/платформы  
7. **сценарии:** 3+ конкретных задач (не функции, а действия)  
8. **техдетали:** если релевантны для доверия (защита данных, модели)  
9. **метрики:** разнотипные (roi, скорость, удовлетворённость, охват)  
10. **перевод метрик:** в понятные единицы (часы = люди)  
11. **честность:** траектория улучшений (было → стало → цель)  
12. **планы:** что дальше (опционально, но усиливает)  
13. **итоги:** обобщение + социальное доказательство  
14. **cta:** привязанный к кейсу

### проверка качества:

☐ заголовок заставляет СЧИТАТЬ (а не просто читать)?  
☐ проблема узнаваема для целевого читателя?  
☐ показан ПРОЦЕСС решения (а не только результат)?  
☐ есть минимум 3 конкретных сценария?  
☐ метрики разнотипные (а не только roi)?  
☐ есть перевод абстрактных метрик в конкретные?  
☐ есть ОДНО честное признание несовершенства?  
☐ есть социальное доказательство?  
☐ каждый блок отвечает на ОДИН вопрос?  
☐ финал оставляет ощущение "это работает в реальности"?

## КРИТИЧЕСКИ ВАЖНО: ЧТО УБИВАЕТ КЕЙС

### смертельные ошибки (никогда не делай):

❌ **абстрактный заголовок** ("повысили эффективность" вместо цифр)  
→ читатель пропускает с первой секунды

❌ **уникальная проблема** ("только у нас было так")  
→ читатель не узнаёт себя, теряет интерес

❌ **решение без процесса** ("мы внедрили ai и всё заработало")  
→ выглядит как магия или вранье

❌ **только функции технологии** ("умеет искать, анализировать, генерировать")  
→ не визуализируется, не запоминается

❌ **одна метрика** (только roi или только скорость)  
→ легко подогнать, не верится

❌ **отсутствие честности** (всё идеально с первого дня)  
→ похоже на рекламу, доверие не формируется

❌ **самовосхваление вместо соцдоказательства** ("мы лучшие на рынке")  
→ маркетинг, а не кейс

❌ **метрики без перевода** (310к часов, без объяснения что это 155 человек)  
→ абстракция не запоминается

❌ **смешивание вопросов в одном блоке** (проблема + решение + метрики в одном абзаце)  
→ читатель теряет нить

❌ **закрытый финал** ("проект завершён, цели достигнуты")  
→ ощущение одноразовой истории, не вдохновляет

## ИТОГОВАЯ ФОРМУЛА

**эталонный кейс = система последовательного закрытия возражений**

где:
- **легитимация** снимает "это вообще серьёзно?"
- **типизация проблемы** снимает "это не про меня"
- **процесс** снимает "это невоспроизводимо"
- **сценарии** снимают "это теория"
- **разнотипные метрики** снимают "цифры подогнаны"
- **честность** снимает "это реклама"
- **социальное доказательство** снимает "только они так думают"

**критично:** последовательность + конкретика + честность  
**гибко:** детализация + тональность + визуализация

воспроизводимость = чеклист выше + понимание функции каждого блока
