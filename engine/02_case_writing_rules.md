# правила работы над кейсами кейсориум

## источники правил

этот документ основан на:
- анализе референсного кейса совкомбанк (`01_reference_case.md`)
- всех правках по кейсу т-банка (итерации v1 → v2 → final)
- контексте кейсориума (`00_context.md`)

**ОБЯЗАТЕЛЬНО:** перед написанием или редактурой кейса также прочитай `engine/wp_layout_rules.md` — правила верстки под WordPress (хлебные крошки, блоки, форматирование, антипаттерны).

## базовая философия

кейс кейсориума = бизнес-документ, а не маркетинговый текст

**критично:**
- конкретика вместо абстракций
- честность про ограничения
- процесс, а не только результат
- применимость для других компаний

**нельзя:**
- додумывать факты и цифры
- использовать клише типа "это не про X, это про Y"
- делать рекламные обещания
- общие слова без метрик

## голос кейса: от лица компании, НЕ от лица спикера

кейс пишется от третьего лица. мы рассказываем историю компании и проекта.

**ЗАПРЕЩЕНО:**
- ссылаться на спикера по имени в тексте: НЕ «Алексей Носов называет это», НЕ «по словам Носова»
- упоминать конференцию, доклад, презентацию как источник: НЕ «как рассказал на конференции»
- давать ссылки на Telegram-каналы, соцсети, внешние ресурсы компании
- писать «подробнее в Telegram-канале @...» или любые внешние CTA

**ПРАВИЛЬНО:**
- «команда называет это» / «в Аспирити называют это»
- «команда поясняет логику» / «разработчики объясняют»
- безличные конструкции: «решение получило название», «подход работает так»

**имя спикера** используется ТОЛЬКО в блоке «Для дизайна:» (поле «Автор:»). в тексте кейса — никогда.

**внешние ссылки** — ТОЛЬКО на саму компанию (сайт), если нужна для контекста. никаких Telegram, соцсетей, подписок.

## структура кейса (канон)

### 1. заголовок
**формат:** Как [Компания] + [действие] + [результат с метрикой]

**примеры:**
- ✓ "Как Т-Банк научил AI-агента работать в топ-3 операторов и автоматизировать 45% обращений"
- ✓ "Как Совкомбанк создал AI-помощника, который экономит 310 000 рабочих часов в год"
- ✗ "Как банк улучшил эффективность с помощью AI" (нет конкретики)

**требования:**
- название компании
- конкретная метрика (не "улучшили", а цифра)
- говорящий (сразу понятно, о чем кейс)

### 2. справка о компании
**что включать:**
- название + позиция на рынке (топ-X, крупнейший)
- масштаб (клиенты, активы, сотрудники)
- чем занимается (кратко)

**пример:**
```
Т-Банк — один из крупнейших цифровых банков России с 53,7 млн клиентов 
и активами под управлением 6,15 трлн рублей. В операционных подразделениях 
работают десятки тысяч сотрудников, которые обрабатывают обращения клиентов, 
ведут продажи и управляют взысканием долгов.
```

### 3. блок для дизайна
**где:** сразу после справки о компании

**что включать:**
- заголовок на карточке (2 строки)
- фильтр (отрасль)
- автор (имя + должность)
- фото (где взять)
- ссылка (url)
- пикчи (список изображений)
- тайтл (seo-заголовок)
- дескрипшен (150-160 символов)
- seo-слова (20-40 штук)

**детали:** смотри `design_block_guide.md`

### 4. TLDR
**формат:** короткое описание о чем кейс + результат в конце

**объем:** 1 абзац, 3-4 предложения

**правило:** НЕ разбивать на «проблема / решение / результат». просто рассказать, о чем кейс, и закрыть результатом.

**пример:**
```
Команда AI-сервисов 2ГИС автоматизирует внутренние процессы — контроль качества 
звонков, классификацию обращений в поддержке, модерацию контента. Кейс о том, 
как команда выстроила подход к внедрению LLM-проектов: от первых прототипов, 
которые не прижились, до работающей системы с пилотами, shadow mode 
и прозрачными метриками. Результат — тысячи сэкономленных человеко-часов 
и чек-лист, с которым AI-проекты доходят до продакшена.
```

### 5. основные блоки

#### контекст / проблема
- почему компании это нужно
- масштаб проблемы
- что уже пробовали

#### решение
- философия подхода (если есть)
- выбор первого домена
- процесс создания (шаги)
- технические детали

#### результаты
- метрики (разнотипные: roi + скорость + качество)
- блок с цифрами для дизайна

#### масштабирование (если есть)
- следующие домены
- проблемы + решения

#### контроль
- как управляют
- безопасность

#### планы
- что дальше
- долгосрочная цель

#### итоги
- обобщение
- вывод для рынка

**важно:** последовательность критична (от контекста к итогам)

## правила написания текста

### предложения: цельные и логичные

**нельзя:**
```
Операторы работают как обычно. Ничего не видят. 
Система записывает логи.
```

**надо:**
```
Операторы работают как обычно и ничего не видят, 
а система записывает логи того, что они делают в интерфейсе.
```

**как объединять:**
- союзы (и, а, но, поэтому, ведь)
- причастные обороты (который, которая)
- деепричастные обороты (при этом)
- вводные конструкции (в результате, например)

### подзаголовки: говорящие

**нельзя:**
```
## Контекст
## Вызов
## Решение
```

**надо:**
```
## Масштаб диктует сложность: почему операционку невозможно просто заморозить
## Философия подхода: общие системы побеждают узкоспециализированные
## Пять шагов от траекторий до автономии
```

**требования:**
- сразу понятно, о чем блок
- не просто тема, а суть
- можно читать только подзаголовки и понять структуру

### жирным ТОЛЬКО числа и метрики

выделяй:
- технические детали с цифрами ("**32 миллиарда** параметров", "**80%** точность")
- метрики ("**45%** трафика", "**98 баллов**")

НЕ выделяй:
- описательные фразы (не "**computer-use агентов**")
- инсайты и философию (не "**общие системы побеждают**")
- подходы и методы (не "**скриптовый домен**")
- справку о компании (НИКОГДА не выделять жирным)

**цель:** жирный = цифры и метрики, остальное — обычным текстом

### списки где уместно

используй списки для:
- перечисления уровней/этапов
- структурированной информации
- сравнения

**пример:**
```
Управление агентами идет на нескольких уровнях:

До релиза: модель тестируют на тестовых данных.

В онлайне: работают гардрейлы — правила, которые анализируют действия агента.

Наследование ограничений: агенты работают с теми же ограничениями, что и люди.
```

### маркеры иллюстраций

если есть презентация спикера, расставляй в тексте маркеры иллюстраций — точные указания, КУДА и КАКОЙ визуал вставить.

**КРИТИЧНО:** НЕ использовать стандартный markdown `![alt](path)` — паблишер его НЕ обрабатывает!

**единственный рабочий формат маркера:**
```
[ИЛЛЮСТРАЦИЯ: слайд #N — описание с визуальным ключевым словом]
```

**обязательные элементы:**
1. номер слайда: `слайд #5` или `слайды #5-6`
2. визуальное ключевое слово (хотя бы одно): `схем`, `скриншот`, `интерфейс`, `архитектур`, `диаграмм`, `роадмап`, `pipeline`, `dashboard`, `CRM`, `демо`, `таблиц`

без этих элементов картинка НЕ загрузится в WordPress.

**где ставить:**
- после абзаца, который маркер иллюстрирует (не до, а ПОСЛЕ)
- перед заголовком раздела (если визуал — обзорная схема блока)
- в начале блока с цифрами (итоговая инфографика)

**типы маркеров:**
- **схемы архитектуры/процесса** → после описания в тексте
- **скриншоты интерфейса** → после описания функции
- **графики метрик** → после упоминания цифр
- **инфографика масштаба** → в начале раздела контекста
- **сравнительные таблицы** → после блока сравнения
- **roadmap** → в разделе планов

**сколько маркеров:**
- минимум 5-7 на кейс (если есть презентация)
- максимум — по одному на каждый смысловой блок
- НИКОГДА не ставить 3+ маркера подряд без текста между ними. максимум 2 подряд (галерея). если есть 3 визуала — раздели текстом

**зачем это нужно:**
- дизайнер видит, куда ставить иллюстрации
- редактор видит, что запрашивать у компании
- текст кейса = готовый макет для верстки

**пример в контексте:**
```
Решением стало дообучение: **T-Pro на 32 миллиарда параметров**. 
Точность на офлайн-метриках — 88%.

[ИЛЛЮСТРАЦИЯ: слайды #21-22 — диаграмма роста точности от ChatGPT к T-Pro 88%]

### Шаг третий: проверить в режиме копилота
```

### цитаты спикера

если источник = доклад или расшифровка интервью, используй прямые цитаты:

**пример:**
```
Как рассказывает руководитель проекта, «наши сотрудники так похожи на людей, 
что ничем не отличаются во всех системах, даже пытались выплатить премию Афанасию, 
начали искать, кому оплатить, и не нашли».
```

### блок с цифрами для дизайна

в итогах создай отдельный блок:

```
### Ключевые результаты проекта

**45%**  
трафика автоматизировано полностью автономно

**40%**  
экономия времени дополнительно к чат-боту

**98 баллов**  
получает AI-агент по 100-балльной системе
```

**формат:**
- цифра жирным
- описание с новой строки
- 5-7 ключевых метрик

### короткий вывод

**нельзя:** 3-4 абзаца с повторами

**надо:** 2 абзаца, сжатый вывод

**что включать:**
- главный инсайт проекта
- как изменился подход
- применимость для других

### ё → е
везде заменяй ё на е (все, еще, учетки)

### кавычки → елочки
"текст" → «текст»

### AI везде большими
ai → AI
ai-агент → AI-агент

### без разделителей
не используй `---` между блоками

## что критично / что гибко

### КРИТИЧНО (нельзя менять):

1. **структура фаз:** легитимация → проблема → решение → результаты → итоги
2. **конкретная метрика в заголовке:** не "улучшили", а цифра
3. **легитимация через масштаб:** топ-X, N клиентов
4. **проблема как типовая для сегмента:** не уникальная
5. **решение как процесс:** шаги, а не только результат
6. **конкретные рабочие сценарии:** не функции, а задачи
7. **честность про несовершенство:** одно признание обязательно
8. **перевод метрик в понятные единицы:** часы = люди
9. **разнотипные метрики:** roi + скорость + качество
10. **TLDR в начале:** проблема → решение → результат
11. **блок для дизайна:** после справки о компании
12. **блок с цифрами:** в итогах
13. **маркеры иллюстраций:** если есть презентация — `[ИЛЛЮСТРАЦИЯ]` в тексте (минимум 5-7)

### ГИБКО (можно адаптировать):

- длина блоков
- степень технических деталей
- количество конкретных сценариев (главное не меньше 3)
- формулировка cta
- тональность (сдержанная ↔ живая, но деловая)

## работа с источниками

### если источник = интервью/доклад:

1. **записывай расшифровку** как есть
2. **не додумывай цифры** — если нет в интервью, пропускай
3. **используй прямые цитаты** спикера
4. **сохраняй последовательность** из доклада
5. **включай все детали** (технические, инсайты, проблемы)

### если чего-то не хватает:

**критичные пропуски** (обозначь явно):
- название компании
- метрика для заголовка
- масштаб компании
- социальное доказательство (премия/награда)

**желательные** (можно оставить без):
- сроки проекта
- размер команды
- точный технический стек

### где искать данные:

**для легитимации:**
- поиск в интернете (клиенты, активы, позиция на рынке)
- официальный сайт компании
- отчеты (если публичная компания)

**для социального доказательства:**
- премии/награды
- упоминания в СМИ
- "один из первых в россии, кто..."

## типичные ошибки

### ❌ обрывистые предложения
```
Операторы работают. Система записывает. Результат — логи.
```
✓ объединяй через союзы и обороты

### ❌ абстрактный заголовок
```
Как банк улучшил эффективность
```
✓ конкретная метрика: "автоматизировал 45% обращений"

### ❌ клише
```
Это не про технологии, это про людей
```
✓ избегай шаблонных конструкций

### ❌ рекламные обещания
```
Наше решение позволит вам увеличить прибыль в 10 раз
```
✓ только факты и метрики из кейса

### ❌ технический подзаголовок
```
## Решение
```
✓ говорящий: "Пять шагов от траекторий до автономии"

### ❌ длинный вывод
3-4 абзаца с повторами  
✓ 2 абзаца, сжато

### ❌ разделители
```
---
## Следующий блок
```
✓ без ---

### ❌ только одна метрика
"Экономия 40%"  
✓ разнотипные: автоматизация + качество + экономия

## примеры применения правил

### пример 1: объединение предложений

**было:**
```
Операторы работают как обычно. Ничего не видят. 
Система записывает логи. Фиксируется состояние экрана.
```

**стало:**
```
Операторы работают как обычно и ничего не видят, а система 
записывает логи того, что они делают в интерфейсе, фиксируя 
состояние экрана, действия и данные в JSON-формате.
```

### пример 2: говорящий подзаголовок

**было:**
```
## Масштабирование
```

**стало:**
```
## Масштабирование на сложные домены: где система дает сбой и как это чинить
```

### пример 3: блок с цифрами

**было:**
```
Результаты: агент автоматизировал 45% трафика, получил 98 баллов, 
экономит 40% времени.
```

**стало:**
```
### Ключевые результаты проекта

**45%**  
трафика автоматизировано полностью автономно

**40%**  
экономия времени дополнительно к чат-боту

**98 баллов**  
получает AI-агент по 100-балльной системе
```

## быстрый старт

1. читай расшифровку интервью/доклада
2. собери метаданные (название, масштаб, автор)
3. найди данные о компании в интернете (если нужно)
4. заполни блок для дизайна
5. напиши TLDR (проблема → решение → результат)
6. пиши основной текст, следуя структуре
7. выдели жирным ключевые мысли
8. создай блок с цифрами в итогах
9. напиши короткий вывод (2 абзаца)
10. проверь по чек-листу
