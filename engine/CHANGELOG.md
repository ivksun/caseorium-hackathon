## февраль 2026: новый формат TLDR

### проблема
старый формат TLDR (проблема → решение → результат) выглядел шаблонно и механически. разбивка на три блока заставляла пересказывать кейс формулами, а не рассказывать о нем.

### решение
новый формат: один абзац на 3-4 предложения. короткое описание, о чем кейс, и в конце — результат. без жесткой структуры «проблема / решение / результат».

### что обновлено
- `02_case_writing_rules.md` — раздел "4. TLDR"
- `00_pipeline_orchestrator.md` — строка про TLDR в канонической структуре
- `CLAUDE.md` — раздел про этап B

### урок
TLDR должен читаться как тизер, а не как пересказ по формуле.

## февраль 2026: визуальная разметка кейсов

### проблема
кейсы писались только как текст. презентации анализировались, но ВИЗУАЛЬНЫЕ элементы (схемы, скриншоты, графики) НЕ фиксировались в тексте кейса. в результате:
- дизайнер не знал, КУДА ставить иллюстрации
- редактор не знал, ЧТО запрашивать у компании
- кейс = стена текста без разметки под визуал

### решение
добавлена система маркеров `[ИЛЛЮСТРАЦИЯ: ...]` — точные указания прямо в тексте кейса, где и какой визуал должен быть.

**что изменилось:**
- `slides_mapping_prompt.md`: добавлена 5-я категория извлечения «визуальные элементы», типы A/B/C (готовый/создать/запросить), секция визуальной карты
- `02_case_writing_rules.md`: новое правило про маркеры иллюстраций, формат, примеры размещения
- `00_pipeline_orchestrator.md`: шаг B4.5 (расстановка маркеров), обновленный output этапа B, пункт в финальный чек-лист

**формат маркера:**
```
`[ИЛЛЮСТРАЦИЯ: описание. Источник: слайд #N презентации. Действие: запросить/перерисовать/создать]`
```

**норма:** минимум 5-7 маркеров на кейс (если есть презентация)

**пример применения:** кейс tbank — 12 маркеров расставлено по тексту

### impact
- кейс = готовый макет для верстки
- дизайнер видит точные места иллюстраций
- редактор знает, что запрашивать у компании
- презентация = источник не только фактуры, но и визуала

## февраль 2026: доработка кейса tbank на основе презентации

### контекст
кейс был написан на основе расшифровки, но презентация спикера (48 слайдов, конференция Conversations 5 декабря 2025) содержала КРИТИЧНЫЕ данные, которых не было в первой версии.

### что обнаружилось

**12 критичных правок из презентации:**

1. **автор**: был «Дмитрий [фамилия уточнить]» → стал **Дмитрий Битман, Product Lead AI центра Т-Банка**
2. **500К строк** датасета в день (было «сотни тысяч» — расплывчато)
3. **88% точность** на офлайн-метриках (было «около 80%» — НЕВЕРНО)
4. **ChatGPT ~30% vs своя модель 50%+** — целый новый слой нарратива про эволюцию точности
5. **T-Pro 32b + SFT** — конкретное название модели (было просто «32 миллиарда параметров»)
6. **50%+ идеальных сессий** — конкретная цифра готовности к автономии
7. **5 Афанасиев в Kubernetes** — техническая конкретика развертывания
8. **до 5 агентов параллельно** на машине оператора — новая проблема + решение
9. **64% acceptance rate** в мобайле (было «60%» — округление искажает)
10. **red teaming** в контроле — было просто «тестируют»
11. **UI vs API** — зачем кликают, а не ходят в API (обоснование подхода из слайда 47)
12. **«нет интентов, нет процедур»** — ключевое отличие от ботов (слайд 48)

**+ добавлено:**
- **3 плашки** с инсайтами
- **Tier 1 / Tier 2** — уровни сложности доменов
- **12 маркеров иллюстраций** по тексту

### урок
презентация НЕ заменяет расшифровку, но ЧАСТО содержит:
- точные цифры (в расшифровке спикер говорит «около», в презентации — точная цифра)
- технические детали (названия моделей, архитектура)
- визуал (схемы, скриншоты, которые КРИТИЧНЫ для понимания)

**правило:** если есть презентация — ВСЕГДА анализировать её через `slides_mapping_prompt.md` и сверять с расшифровкой. презентация ДОПОЛНЯЕТ расшифровку и даёт визуальную карту.

## ключевые принципы работы (закреплены)

### 1. презентация = источник фактуры + визуала
- фактура: цифры, формулировки, технические детали
- визуал: схемы, скриншоты, графики → маркеры `[ИЛЛЮСТРАЦИЯ]` в тексте

### 2. маркеры иллюстраций = обязательная часть кейса
если есть презентация — минимум 5-7 маркеров по тексту. маркер = точное место + описание + источник + действие.

### 3. не додумывать, сверять с источниками
валидация фактов (`fact_validation_prompt.md`) — обязательный этап. каждая цифра, формулировка, вывод проверяются на соответствие расшифровке/презентации.

### 4. технические правила строгие
- ё → е везде
- кавычки → елочки «»
- AI большими буквами
- нет разделителей ---
- предложения цельные (союзы, обороты)

### 5. pipeline = автономный процесс
A → B → C → D → E без остановок (кроме критичных пропусков данных). не спрашивать разрешения между этапами.

## типовые проблемы и решения

### проблема: «около 80%» vs «88%»
**что было:** расшифровка говорит «около 80%», кейс пишет «около 80%»
**что на самом деле:** в презентации точная цифра — 88%
**решение:** презентация = приоритет для точных цифр

### проблема: обобщение вместо конкретики
**что было:** «сотни тысяч строк датасета»
**что на самом деле:** 500К строк каждый день
**решение:** если в презентации есть конкретная цифра — использовать её

### проблема: отсутствие визуальной разметки
**что было:** кейс = текст, визуал решали post factum
**что теперь:** маркеры `[ИЛЛЮСТРАЦИЯ]` прямо в тексте кейса на этапе сборки

### проблема: утеря технических деталей
**что было:** «модель на 32 миллиарда параметров»
**что на самом деле:** «T-Pro 32b + доменная адаптация + SFT»
**решение:** презентация часто содержит точные названия технологий
