# Case Builder — пайплайн Кейсориума

## Правила и стиль

Перед запуском прочитай:
- `references/russian-style-guide.md` — инфостиль + антинейросетевые правила
- `engine/02_case_writing_rules.md` — структура, ToV, чек-лист
- `engine/design_block_guide.md` — блок для дизайна
- `engine/00_context.md` — для кого пишем, зачем

Эти файлы = источник правды. Не импровизируй.

---

## Pipeline: 4 шага

### ШАГ 1 — Аудит входных данных

**1а. Подготовка слайдов (ПЕРВЫЙ ШАГ — ДО ВСЕГО ОСТАЛЬНОГО)**

Перед тем как читать расшифровку и писать кейс — проверь, есть ли PDF-презентация:

1. **Проверь вложения в сообщении пользователя.** Если в контексте есть `<document>` с PDF (тег `<source>` содержит `.pdf`) — это презентация. Извлеки имя файла из `<source>`.
2. **Проверь, упоминал ли пользователь PDF, слайды, презентацию** в тексте сообщения.
3. **Если пользователь дал YouTube-ссылку** — презентация часто идет в комплекте. Спроси: «Есть ли PDF презентации?» если не приложен.

**Если PDF найден (вложение, файл или ссылка) — ОБЯЗАТЕЛЬНО сохрани и извлеки слайды. Не продолжай без этого.**

**Как найти файл на диске:**

PDF-вложение из чата всегда существует как файл. Найди его:
```bash
# Ищи по имени из <source> тега вложения (например, "05-Krestnikov.pdf")
find ~/Desktop ~/Downloads /tmp . -name "<имя_файла>.pdf" -maxdepth 3 2>/dev/null | head -10
```
Также проверь рабочую директорию проекта. Если не нашел — спроси у пользователя путь. **НЕ пиши «слайды не загружены» и не продолжай без файла.**

**Шаг 2: Сохранить и извлечь.**

```bash
mkdir -p cases/<company>/slides/
cp <найденный_путь.pdf> cases/<company>/presentation.pdf
python3 tools/extract_slides.py cases/<company>/presentation.pdf cases/<company>/slides/
```

**Шаг 3: Просмотреть слайды визуально.**

Используй Read tool на PNG-файлах (`cases/<company>/slides/slide_01.png` и т.д.). Составь список:
- Какие слайды содержат визуал (скриншоты, схемы, диаграммы, интерфейсы, таблицы)
- Какие слайды текстовые (пропускаем)
- Запиши номера визуальных слайдов — они понадобятся на ШАГе 2

**НЕ ПРОПУСКАЙ ЭТОТ ШАГ.** Без файла слайдов на диске маркеры `[ИЛЛЮСТРАЦИЯ]` бесполезны — паблишер не сможет загрузить картинки. Писать маркеры «вслепую» по тексту PDF и надеяться, что кто-то потом добавит картинки вручную — запрещено.

**1б. Аудит расшифровки на транскрипционный мусор**

Расшифровки часто содержат артефакты: слова, которые похожи на ошибки распознавания — неправильно переданные имена, термины, аббревиатуры, числа, обрезанные фразы.

Пройди по расшифровке и выдели подозрительные места:
- Слово, которое не подходит по смыслу в контексте
- Название/термин, который выглядит искажённым (например, "айпроп" вместо "AI prop")
- Число, которое кажется нереальным или обрезанным
- Фраза, которая логически не достраивается

Для каждого: сформулируй наиболее вероятную интерпретацию и отметь `[? уточнить]`.
Если уверен — исправь сам и иди дальше.

**1в. Стратегический разбор**

Сформулируй для себя (и скажи мне):
- **Главный инсайт кейса** — что в нём реально нового или неочевидного для рынка?
- **Для какой аудитории он сильнее всего** — ориентир: аудитория кейсориума из `00_context.md`, не пытайся определить из расшифровки
- **Что делает этот кейс уникальным** — подход, масштаб, честность про ограничения?
- **Что кажется подозрительным** — места, которые надо уточнить у спикера

Только то, что спикер буквально говорил. Никаких домыслов и интерпретаций.

Выдай стратегический разбор коротко — 4-6 строк. Жди подтверждения или правок от меня.

### ШАГ 2 — Сборка черновика (в памяти)

Используй `02_case_writing_rules.md`.

Пиши кейс по канонической структуре:

1. **Заголовок** — `Как [Компания] + [действие] + [результат с метрикой]`
2. **Справка о компании** — позиция на рынке, масштаб, чем занимается
3. **Блок для дизайна** — 9 полей по `design_block_guide.md`
4. **TLDR** — структура: [описание сути кейса 1-2 предложения] + отдельная строка «Результат: [метрики]». Коротко, результат выносить явно.
5. **Основные блоки** — контекст → решение → результаты → масштабирование → контроль → планы → итоги

Правила:
- Только факты из расшифровки — не додумывать
- Если данных нет — `[уточнить]`, продолжай
- Если нет масштаба компании — поищи в интернете
- Цитаты спикера — в кавычках «»
- Уровни заголовков: секции кейса = `##` (включая «Ключевые результаты проекта»), внутри :::блоков = `###`

**Иллюстрации из слайдов:**

На ШАГе 1а ты уже просмотрел все слайды и составил список визуальных. Теперь:

1. **Для каждого визуального слайда** из списка определи, к какому абзацу кейса он относится по смыслу.
2. **Поставь маркер ПОСЛЕ абзаца**, который слайд иллюстрирует.
3. Формат маркера (единственный рабочий, другие НЕ работают):
   ```
   [ИЛЛЮСТРАЦИЯ: слайд #N — описание с визуальным ключевым словом]
   ```
4. **Обязательные элементы** (без любого из них картинка НЕ загрузится в WordPress):
   - Номер слайда: `слайд #11` или `слайды #5-6`
   - Хотя бы одно ключевое слово: `схем`, `скриншот`, `интерфейс`, `архитектур`, `диаграмм`, `роадмап`, `pipeline`, `dashboard`, `CRM`, `демо`, `таблиц`
5. **ЗАПРЕЩЕНО** `![alt](path)` — паблишер НЕ обрабатывает markdown-картинки.
6. Минимум **5-7 маркеров** на кейс. Максимум 2 подряд без текста.

Примеры:
```
[ИЛЛЮСТРАЦИЯ: слайд #11 — скриншот интерфейса MVP для контроля качества звонков]
[ИЛЛЮСТРАЦИЯ: слайды #5-6 — схема архитектуры пайплайна обработки данных]
```

### ШАГ 3 — Редактура (один проход, в памяти)

Один проход вместо двух отдельных этапов. Делай всё параллельно:

**Факты:**
- Цифры совпадают с расшифровкой
- Нет интерпретаций под видом фактов («революционный подход» → убрать)
- Нет выводов без опоры на данные
- Подозрительные места из Шага 1 — проверены или помечены `[уточнить у спикера]`

**Логика:**
- Причинно-следственные связи чёткие (не «решили использовать AI», а «выбрали AI, потому что чат-боты закрывали только часть задач»)
- Переходы между блоками объяснены
- Переход от пилота к масштабированию — явный

**Текст:**
- Убрать воду: «в современном мире», «как известно», «стоит отметить»
- Убрать клише: «это не про X, это про Y»
- Убрать излишние противопоставления «не X, а Y» — если это моя конструкция, а не слова спикера
- Оборот «по словам спикера...» — не использовать. Либо прямая цитата «...» — Имя, должность. Либо просто убрать оборот.
- Отраслевые термины (медпред, девопс, мерч и т.п.) — проверить: понятны ли читателю вне этой отрасли? Если нет — заменить на расшифровку.
- Объединить обрывистые предложения через союзы и обороты
- Сжать без потери смысла

---

### ШАГ 4 — Финальный формат + грамматика

**4а. Форматирование**
- ё → е везде
- Кавычки → «елочки»
- AI везде заглавными (ai-агент → AI-агент)
- Убрать разделители `---`
- Жирным: цифры, технические детали, ключевые инсайты, философские тезисы — НО ТОЛЬКО в обычном тексте (custom_text). ЗАПРЕЩЕНО внутри :::columns, :::steps, :::list — звездочки рендерятся буквально
- 2-3 плашки с сильными инсайтами — только из прямых слов спикера (слайды или расшифровка), не добавлять свои формулировки
- Итоги — максимально короткие: строго по тому, что спикер сам формулировал как выводы. 2-3 предложения.
- Блок «Ключевые результаты проекта» в итогах (5-7 метрик)

**4б. Грамматика и пунктуация — отдельный проход**

Пройди по тексту явно с фокусом на:
- Запятые в сложносочинённых и сложноподчинённых предложениях
- Причастные и деепричастные обороты — запятые расставлены верно?
- Однородные члены — запятые между ними
- Вводные конструкции («например», «в результате», «при этом») — выделены запятыми?
- Нет лишних тире вместо запятых
- Числа и единицы — без пробелов где не нужно, с неразрывным пробелом где нужно

Это явный шаг, не просто пункт в чек-листе.

**4в. Чек-лист**

Пройди по чек-листу из `02_case_writing_rules.md`. Если что-то не выполнено — исправь сейчас, не отмечай как ОК.

## Как запустить

### Вариант 1: YouTube + имя спикера (полная автоматизация)
Скажи: «собери кейс» и укажи:
- YouTube URL с записью доклада
- Имя спикера (для поиска презентации на Google Drive)
- Опционально: имя компании

Pipeline автоматически:
1. Транскрибирует видео (ШАГ 0)
2. Найдёт и скачает PDF презентации с Google Drive (ШАГ 0.5)
3. Соберёт кейс (ШАГ 1-4)
4. Опционально: опубликует черновик в WordPress (ШАГ 5)

### Вариант 2: Готовая расшифровка (как раньше)
Скажи: «собери кейс» — и прикрепи расшифровку (+ слайды опционально).

## Предварительные этапы (автоматизация)

### ШАГ 0 — Транскрипция YouTube (если вход = URL)

Если пользователь дал YouTube-ссылку вместо расшифровки:

1. Запусти скрипт транскрипции:
   ```bash
   python3 tools/transcribe_youtube.py "<URL>" --output cases/<company>_draft/ --method deepgram
   ```
   Если Deepgram недоступен — fallback на `--method youtube`.

2. Результат: `cases/<company>_draft/transcript.md`
3. Используй этот файл как входную расшифровку для ШАГа 1.

### ШАГ 0.5 — Презентация с Google Drive (если указан спикер)

Если пользователь указал имя спикера:

1. Через Google Drive MCP найди PDF по имени: ищи `*<Фамилия>*` в папке "Презентации Conversations"
2. Скачай PDF → `cases/<company>_draft/presentation.pdf`
3. Извлеки слайды:
   ```bash
   python3 tools/extract_slides.py cases/<company>_draft/presentation.pdf
   ```
4. Результат: `cases/<company>_draft/slides/slide_01.png` ... `slide_NN.png`
5. Используй PDF и слайды для анализа на ШАГе 1.

---

## Постобработка (автоматизация)

### ШАГ 5 — Публикация в WordPress (опционально)

После создания `_READY.md`, если пользователь попросил опубликовать:

1. Запусти паблишер **с указанием папки слайдов** (если слайды есть):
   ```bash
   python3 tools/publish_to_wp_v2.py cases/<company>/<company>_READY.md --slides-dir cases/<company>/slides/
   ```
   Без `--slides-dir` картинки НЕ загрузятся, даже если маркеры `[ИЛЛЮСТРАЦИЯ]` расставлены.

2. **Как это работает:**
   - Паблишер находит маркеры `[ИЛЛЮСТРАЦИЯ: слайд #N — ...]` в тексте
   - По номеру слайда ищет файл `slide_NN.png` в `--slides-dir`
   - Загружает PNG в медиабиблиотеку WordPress
   - Создает ACF-блок `layer_media` на месте маркера
   - Маркер удаляется из текста

3. Скрипт создаст **черновик** (draft) в WordPress с заполненными ACF-полями и загруженными картинками
4. Выдай пользователю ссылку на редактирование в WP-админке
5. Напомни: проверить картинки, ACF-поля, добавить превью-картинку

**Не публикуй сразу** — только черновик. Финальную публикацию делает человек.

**ВАЖНО:** если на ШАГе 1а слайды были извлечены в `cases/<company>/slides/`, то на ШАГе 5 ОБЯЗАТЕЛЬНО передай `--slides-dir cases/<company>/slides/`. Иначе маркеры иллюстраций останутся пустыми.
