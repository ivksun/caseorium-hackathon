## Agent Platform

---

### сценарий 1: AI-рекрутер (автоскрининг)

**тип клиента**: enterprise с массовым наймом (промышленность, ритейл, логистика, банки)
**проблема**: нехватка рекрутёров для обработки большого потока кандидатов; медленный первый контакт; рутинный скрининг занимает 70%+ времени рекрутёра
**контекст**: компания пропускает миллионы кандидатов через воронку найма на заводы / в розницу / в колл-центры
**как внедряют**:
- AI-агент сам звонит кандидатам (холодный дозвон)
- ведёт гибкую беседу, задаёт скрининговые вопросы
- фиксирует ответы, ставит авто-вердикт
- переводит кандидата по воронке в HRIS/ATS
- рекрутёр подключается только к финалистам

**ограничения**:
- требует интеграции с ATS/HRIS клиента
- качество зависит от скрипта и настройки voice-модели
- не заменяет глубинное интервью, только первичный скрининг

**измеримые результаты**:
- сокращение времени до первого контакта (из дней -- в часы)
- разгрузка рекрутёров от рутинного обзвона
- увеличение пропускной способности воронки

**каналы**: голос (телефония)
**технологии**: Agent Platform + ASR/TTS + HRIS-интеграция

### сценарий 2: AI-агент для клиентской поддержки

**тип клиента**: банки, телеком, ритейл, страхование -- любой enterprise с большим объёмом обращений в поддержку
**проблема**: высокая нагрузка на команду поддержки, низкий FCR (first contact resolution), долгое время ответа, дорогие операторы
**контекст**: тысячи обращений в день через чат, голос, email; типовые вопросы занимают 60-80% потока
**как внедряют**:
- AI-агент отвечает из базы знаний (RAG через Jay Knowledge Hub)
- классифицирует и маршрутизирует обращения
- подсказывает оператору "next best answer" (суфлёр)
- при необходимости создаёт тикет в ITSM/CRM
- сложные кейсы эскалирует на человека

**ограничения**:
- качество RAG зависит от полноты и актуальности базы знаний
- для голосового канала нужна настройка ASR/TTS
- требует интеграции с ITSM/CRM клиента

**измеримые результаты**:
- увеличение FCR
- снижение нагрузки на команду поддержки (цель: 70% обращений обрабатываются автоматически)
- сокращение среднего времени обработки

**каналы**: чат, голос, email
**технологии**: Agent Platform + Jay Knowledge Hub (RAG) + ITSM/CRM интеграция + голосовые технологии

### сценарий 3: AI-агент для закупок

**тип клиента**: enterprise с масштабными закупочными процессами (промышленность, ритейл, госсектор)
**проблема**: длинный цикл закупки, ручные ошибки в формировании заказов, медленное согласование
**контекст**: десятки/сотни заявок на закупку в день, ручное уточнение требований, подбор позиций и поставщиков
**как внедряют**:
- AI-агент принимает заявки (чат/email)
- уточняет требования в диалоге
- подбирает позиции и поставщиков из каталога
- формирует черновик заказа
- маршрутизирует согласование в ERP/SRM

**ограничения**:
- требует глубокой интеграции с ERP/SRM клиента
- критичен справочник поставщиков и номенклатура
- финальное решение всегда за человеком

**измеримые результаты**:
- сокращение цикла закупки
- снижение ручных ошибок
- ускорение согласования

**каналы**: чат, email
**технологии**: Agent Platform + ERP/SRM интеграция

### сценарий 4: автоматизация бэк-офиса (бухгалтерия, operations)

**тип клиента**: enterprise любой отрасли с большим бэк-офисом
**проблема**: рутинные процессы в бухгалтерии, документообороте, внутренних запросах; сотрудники тратят время на повторяющиеся задачи
**контекст**: упоминается в стратегии продаж как типовой кейс для демо-стендов и расчётов ROI
**как внедряют**:
- типовые AI-агенты для обработки внутренних запросов
- автоматизация документооборота через RAG + workflow
- интеграция с внутренними системами (1С, ERP, корпоративный портал)

**ограничения**:
- требует маппинга конкретных бизнес-процессов
- ROI зависит от объёма рутины

**измеримые результаты**:
- сокращение расходов на обработку типовых запросов
- ускорение внутренних процессов

### сценарий 5: автоматизация лидогенерации для продаж

**тип клиента**: enterprise B2B с активными продажами
**проблема**: дорогой ручной outbound, низкая конверсия холодных контактов, медленная квалификация лидов
**контекст**: упоминается в транскрипте как один из бизнес-кейсов для агентской платформы
**как внедряют**:
- AI-агент для первичной квалификации входящих/исходящих контактов
- автоматический outreach (звонки, email)
- сбор информации и маршрутизация в CRM

**ограничения**:
- требует CRM-интеграции
- tone of voice и качество скриптов критичны

**измеримые результаты**:
- увеличение количества квалифицированных лидов
- сокращение стоимости лида
- ускорение цикла продажи

## Jay Copilot

---

### сценарий 6: корпоративный AI-ассистент для сотрудников

**тип клиента**: средний и крупный бизнес, любая отрасль
**проблема**: сотрудники хаотично используют ChatGPT (через VPN, личные аккаунты), нет контроля, нет аналитики, есть риск утечки данных
**контекст**: компания хочет централизованно предоставить доступ к GenAI, контролировать использование, собирать кейсы
**как внедряют**:
- разворачивают Jay Copilot (cloud или on-prem)
- подключают нужные LLM (GigaChat, YandexGPT, ChatGPT и др.)
- подключают Jay Guard для контроля данных
- создают AI-приложения под задачи отделов (маркетинг: генерация текстов; HR: скрининг резюме; юристы: анализ договоров)
- загружают корпоративные документы в базу знаний

**ограничения**:
- не заменяет глубокую автоматизацию процессов (для этого Agent Platform)
- эффективность зависит от вовлечённости сотрудников и качества AI-приложений

**измеримые результаты**:
- +30% к эффективности сотрудников (заявленная метрика)
- контроль использования GenAI в компании
- сбор реальных сценариев для масштабирования

**роли пользователей**: маркетологи, копирайтеры, дизайнеры, аналитики, HR, рекрутеры, разработчики, юристы, продакт-менеджеры

### сценарий 7: корпоративная база знаний с AI-ответами

**тип клиента**: enterprise с большим объёмом внутренней документации
**проблема**: сотрудники не могут быстро найти нужную информацию в корпоративных документах, тратят часы на поиск
**контекст**: инструкции, регламенты, политики, FAQ -- всё разрозненно и плохо структурировано
**как внедряют**:
- загружают документы в Jay Knowledge Hub (через Jay Copilot)
- сотрудники задают вопросы на естественном языке
- система генерирует ответы с указанием источника

**ограничения**:
- качество зависит от качества документов
- нужна регулярная актуализация базы знаний

**измеримые результаты**:
- сокращение времени поиска информации
- снижение нагрузки на HR/операционный отдел по типовым вопросам

## Jay Guard

---

### сценарий 8: контроль данных в корпоративном AI-ассистенте

**тип клиента**: регулируемый enterprise (банки, страхование, телеком, госсектор)
**проблема**: сотрудники отправляют ПДн клиентов, коммерческую тайну, чувствительные данные в облачные LLM через корпоративный AI-ассистент
**контекст**: компания внедрила Jay Copilot или стороннее решение, но нет контроля, что уходит в нейросеть
**как внедряют**:
- устанавливают Jay Guard как шлюз между пользователями и LLM
- настраивают политики фильтрации (какие данные маскировать, какие блокировать)
- подключают журналирование и алерты

**ограничения**:
- ложноположительные срабатывания возможны при агрессивных политиках
- требует настройки под специфику данных компании

**измеримые результаты**:
- ноль утечек корпоративных данных в нейросети (при правильной настройке)
- соответствие регуляторным требованиям

### сценарий 9: защита AI-агентов от prompt injection и jailbreak

**тип клиента**: enterprise, использующий AI-агентов в production (клиентский сервис, внутренние процессы)
**проблема**: AI-агент может быть "взломан" через prompt injection -- заставлен выдать конфиденциальные данные, выйти за рамки сценария, выполнить вредоносные действия
**контекст**: AI-агент общается с внешними пользователями (клиентами) -- поверхность атаки максимальна
**как внедряют**:
- Jay Guard анализирует входящие запросы к агенту на наличие инъекций
- блокирует или модифицирует подозрительные запросы
- контролирует ответы агента на соответствие политикам

**ограничения**:
- 100% защита невозможна -- гонка вооружений между атакующими и защитой
- требует регулярного обновления моделей детекции

**измеримые результаты**:
- снижение количества успешных prompt injection атак
- автоматическое логирование инцидентов

### сценарий 10: безопасность агентских цепочек действий

**тип клиента**: технологические компании, enterprise с мультиагентными системами
**проблема**: AI-агент выполняет цепочку действий (tool calling) -- каждый шаг расширяет поверхность атаки, агент может выполнить небезопасное действие
**контекст**: эволюция use-case Jay Guard в 2026 -- от маскирования ПДн к управлению поведением LLM и безопасности агентов
**как внедряют**:
- Jay Guard контролирует не только данные, но и действия агента
- политики на уровне цепочек рассуждений и tool calls

**ограничения**:
- новое направление, продукт в активном развитии
- требует тесной интеграции с Agent Platform

**измеримые результаты**:
- предотвращение небезопасных автономных действий
- аудит всех действий агента

### сценарий 11: отраслевое решение "под ключ"

**тип клиента**: enterprise из приоритетных отраслей (банки, телеком, ритейл, промышленность, страхование)
**проблема**: клиент приходит с "чистым листом" -- не знает, с чего начать AI-трансформацию
**контекст**: стратегия продаж предполагает проактивный подход -- приходить с готовыми отраслевыми кейсами и расчётами ROI
**как внедряют**:
- стартовый пакет (5-10 млн, 2-3 мес): один пилотный кейс на одной бизнес-функции
- бизнес-аналитик помогает клиенту сформулировать требования (без доп. оплаты на пресейле)
- используются типовые ТЗ и шаблоны из библиотеки
- облачные демо-стенды для самостоятельного тестирования
- при успехе -- масштабирование на другие департаменты

**ограничения**:
- полноценный пилот с боевым трафиком всегда требует кастомной интеграции
- облачные стенды -- прототипы, не замена полноценного внедрения

**измеримые результаты**:
- time-to-value <90 дней от подписания до первых KPI
- целевой % автоматизации по пилотной функции
- экономический эффект в рублях

### сценарий 12: облачный SaaS для тестирования (cloud + бесплатный дистрибутив)

**тип клиента**: проджекты / продакты внутри enterprise, которые ищут инструмент для прототипа
**проблема**: хочется быстро попробовать агентскую платформу без долгого согласования закупки
**контекст**: во 2 квартале 2026 компания выпускает бесплатный дистрибутив Agent Platform (контейнер для self-deploy)
**как внедряют**:
- скачивают контейнер, разворачивают у себя (Yandex Cloud и т.д.)
- тестируют по демо-ключу с ограничениями
- если подходит -- инициируют закупку enterprise-лицензии

**ограничения**:
- ограниченная функциональность в бесплатной версии
- нет корпоративной поддержки в бесплатном режиме

**измеримые результаты**:
- конверсия из бесплатного тестирования в enterprise-контракт
